## 一、总则

### 1.1 目的

本规范旨在建立统一的安装过程异常处理标准，确保：
1. 异常场景识别完整、分类清晰
2. 异常处理策略一致、可预测
3. 重试机制合理、可控
4. 日志记录规范、可追溯
5. 故障恢复自动化、可人工介入

### 1.2 适用范围

本规范适用于以下安装Phase：
- EnsureCerts（证书生成）
- EnsureBKEAgent（Agent推送）
- EnsureNodesEnv（节点环境初始化）
- EnsureClusterAPIObj（Cluster API对象创建）
- EnsureMasterInit（Master初始化）
- EnsureMasterJoin（Master加入）
- EnsureWorkerJoin（Worker加入）
- EnsureNodesPostProcess（节点后置处理）

### 1.3 规范等级

| 等级 | 标识 | 说明 | 违反后果 |
|------|------|------|---------|
| **强制** | [MUST] | 必须遵守，无例外 | 代码审核不通过 |
| **推荐** | [SHOULD] | 强烈建议遵守，特殊情况可例外 | 需说明理由 |
| **可选** | [MAY] | 可选择遵守 | 无 |

### 1.4 术语定义

| 术语 | 定义 |
|------|------|
| **Fail-Fast** | 遇到错误立即中断流程，不自动重试，需人工介入 |
| **Best-Effort** | 遇到错误记录失败，继续处理其他资源，最后统一报告 |
| **Requeue** | Controller重新将请求加入工作队列，触发再次协调 |
| **BackoffLimit** | 命令执行失败时的最大重试次数 |
| **ActiveDeadlineSecond** | 命令执行的最大允许时间（秒） |
| **Phase** | 安装流程中的一个逻辑阶段 |
| **Condition** | 命令执行状态的详细记录 |

## 二、异常分类规范

### 2.1 异常严重程度分类

**[MUST]** 所有异常必须按照以下标准分类：

| 严重程度 | 定义 | 处理原则 | 日志级别 | 事件类型 |
|---------|------|---------|---------|---------|
| **Fatal** | 导致安装流程无法继续，需人工介入 | Fail-Fast，立即中断 | Error | Warning |
| **Error** | 当前操作失败，但可重试或跳过 | 根据策略重试或标记失败 | Error | Warning |
| **Warning** | 非预期情况，但不影响主流程 | 记录日志，继续执行 | Warn | Normal |
| **Info** | 正常流程中的关键节点 | 记录日志，继续执行 | Info | Normal |

**[MUST]** Fatal级别异常必须满足以下条件之一：
1. Master节点初始化失败
2. Etcd集群不可用
3. 控制平面组件无法启动
4. 证书生成失败且无法恢复
5. 关键配置错误导致无法继续

### 2.2 异常来源分类

**[MUST]** 异常必须明确来源分类：

| 来源 | 说明 | 典型异常 | 默认处理策略 |
|------|------|---------|-------------|
| **Infrastructure** | K8s API、网络、存储等底层异常 | API Server不可达、Secret不存在 | Requeue重试 |
| **Node** | 节点层面异常 | SSH连接失败、Agent未就绪、节点NotReady | 单节点重试 |
| **Cluster** | 集群层面异常 | Etcd不健康、控制平面不可用 | 等待轮询或Fail-Fast |
| **Configuration** | 配置错误或缺失 | BKEConfig无效、版本不支持 | 返回错误，需人工修正 |
| **Command** | Agent命令执行失败 | Kubeadm执行失败、脚本执行错误 | 根据BackoffLimit重试 |

### 2.3 异常场景完整性要求

**[MUST]** 每个Phase必须识别并文档化以下异常场景：
1. 前置条件不满足的场景
2. 资源获取失败的场景
3. 核心逻辑执行失败的场景
4. 状态持久化失败的场景
5. 超时场景
6. 配置错误场景

**[SHOULD]** 每个异常场景应包含：
- 触发条件
- 严重程度
- 处理策略
- 重试机制
- 日志输出
- 人工介入条件

## 三、异常处理策略规范

### 3.1 Phase处理策略选择

**[MUST]** Phase处理策略必须按照以下规则选择：

```
┌─────────────────────────────────────────────────────────────┐
│  Phase处理策略选择规则                                       │
└─────────────────────────────────────────────────────────────┘

IF Phase IN {EnsureMasterInit, EnsureEtcdUpgrade} THEN
    策略 = Fail-Fast
    理由 = Master/Etcd是集群核心，失败必须人工介入
    
ELSE IF Phase IN {EnsureWorkerJoin, EnsureWorkerUpgrade} THEN
    策略 = Best-Effort
    理由 = Worker失败不影响集群核心功能
    
ELSE IF Phase IN {EnsureCerts, EnsureClusterAPIObj} THEN
    策略 = Requeue-Until-Success
    理由 = 临时性错误，可自动恢复
    
ELSE IF Phase IN {EnsureBKEAgent, EnsureNodesEnv} THEN
    策略 = Per-Node-Retry
    理由 = 单节点失败不影响其他节点
    
ELSE IF Phase = EnsureNodesPostProcess THEN
    策略 = Ignore-And-Continue
    理由 = 非关键操作，失败不影响集群功能
END IF
```

### 3.2 Fail-Fast策略规范

**[MUST]** 采用Fail-Fast策略时必须遵守：

```go
// 标准Fail-Fast处理模板
func (e *Phase) Execute() (ctrl.Result, error) {
    // 执行操作
    err := e.doOperation()
    
    if err != nil {
        // [MUST] 立即标记失败状态
        e.markFailed(err)
        
        // [MUST] 持久化状态
        if syncErr := e.syncStatus(); syncErr != nil {
            return ctrl.Result{}, syncErr
        }
        
        // [MUST] 记录Error级别日志
        log.Error(constant.PhaseFailedReason, 
            "Phase failed, need manual intervention: %v", err)
        
        // [MUST] 返回错误，不Requeue
        return ctrl.Result{}, err
    }
    
    return ctrl.Result{}, nil
}
```

**[MUST]** Fail-Fast场景下：
1. 不设置BackoffLimit（或设置为0）
2. 不执行自动重试
3. 必须记录人工介入指引日志
4. 必须标记节点/集群状态为Failed

### 3.3 Best-Effort策略规范

**[MUST]** 采用Best-Effort策略时必须遵守：

```go
// 标准Best-Effort处理模板
func (e *Phase) Execute() (ctrl.Result, error) {
    var failedResources []string
    
    for _, resource := range resources {
        // 执行操作
        err := e.doOperation(resource)
        
        if err != nil {
            // [MUST] 记录失败资源
            failedResources = append(failedResources, resource.ID)
            
            // [MUST] 标记单个资源失败
            e.markResourceFailed(resource, err)
            
            // [MUST] 记录Warning级别日志
            log.Warn(constant.ResourceFailedReason,
                "Resource %s failed: %v, continuing with others", 
                resource.ID, err)
            
            // [MUST] 继续处理下一个资源
            continue
        }
        
        // 标记成功
        e.markResourceSuccess(resource)
    }
    
    // [MUST] 持久化所有状态
    if err := e.syncStatus(); err != nil {
        return ctrl.Result{}, err
    }
    
    // [MUST] 如有失败资源，返回错误触发Requeue
    if len(failedResources) > 0 {
        return ctrl.Result{}, errors.Errorf(
            "some resources failed: %v", failedResources)
    }
    
    return ctrl.Result{}, nil
}
```

### 3.4 Requeue策略规范

**[MUST]** Requeue策略必须明确区分场景：

```go
// Requeue决策规范
func (e *Phase) handleError(err error) (ctrl.Result, error) {
    // [MUST] 资源不存在不Requeue
    if apierrors.IsNotFound(err) {
        return ctrl.Result{}, nil
    }
    
    // [MUST] 版本冲突立即Requeue
    if apierrors.IsConflict(err) {
        return ctrl.Result{Requeue: true}, nil
    }
    
    // [MUST] 临时性错误延迟Requeue
    if isTemporaryError(err) {
        return ctrl.Result{RequeueAfter: quickRequeueInterval}, nil
    }
    
    // [MUST] 其他错误返回错误
    return ctrl.Result{}, err
}
```

**[SHOULD]** Requeue间隔应遵循：
- 快速重试：10秒（临时性错误）
- 中速重试：30秒（资源暂时不可用）
- 慢速重试：5分钟（周期性检查）

## 四、重试机制规范

### 4.1 重试配置规范

**[MUST]** Command CRD重试配置必须明确设置：

```yaml
spec:
  # [MUST] 明确设置最大重试次数
  backoffLimit: <n>  # n >= 0
  
  # [MUST] 明确设置执行超时
  activeDeadlineSecond: <t>  # t > 0
  
  # [SHOULD] 设置完成后保留时间
  ttlSecondsAfterFinished: <ttl>  # ttl >= 0
  
  commands:
    - id: "<unique-id>"
      # [MUST] 每个命令明确是否跳过失败
      backoffIgnore: <true|false>
      
      # [SHOULD] 设置重试间隔
      backoffDelay: <d>  # d >= 0
```

### 4.2 重试次数规范

**[MUST]** 重试次数必须根据场景合理设置：

| 场景 | BackoffLimit | 理由 |
|------|-------------|------|
| **Master初始化** | 0 | Fail-Fast，不重试 |
| **Master加入** | 3 | 允许有限重试 |
| **Worker加入** | 3 | 允许有限重试 |
| **环境初始化** | 3 | 允许有限重试 |
| **Agent推送** | 3 | 允许有限重试 |
| **非关键操作** | 0-1 | 快速失败或跳过 |

**[MUST]** 重试次数不得超过5次，超过需人工介入。

### 4.3 重试间隔规范

**[MUST]** 重试间隔必须合理设置：

| 场景 | BackoffDelay | 理由 |
|------|-------------|------|
| **网络操作** | 5-10秒 | 等待网络恢复 |
| **服务启动** | 3-5秒 | 等待服务就绪 |
| **资源创建** | 2-3秒 | 快速重试 |
| **配置生效** | 1-2秒 | 配置快速生效 |

**[SHOULD]** 重试间隔应考虑：
- 操作类型（网络、磁盘、CPU）
- 资源规模（单节点、多节点）
- 环境状况（生产、测试）

### 4.4 超时配置规范

**[MUST]** 超时时间必须根据操作复杂度设置：

| 操作类型 | ActiveDeadlineSecond | 计算依据 |
|---------|---------------------|---------|
| **证书生成** | 300（5分钟） | CPU密集型 |
| **Agent推送** | 600（10分钟） | 网络传输+服务启动 |
| **环境初始化** | 600-1200（10-20分钟） | 依赖安装+配置 |
| **Master初始化** | 900（15分钟） | Kubeadm init+组件启动 |
| **节点加入** | 600（10分钟） | Kubeadm join+证书复制 |
| **后置处理** | 300（5分钟） | 标签/污点设置 |

**[MUST]** 超时时间计算公式：

```
ActiveDeadlineSecond = 基础时间 + (节点数 × 单节点时间) + 网络延迟补偿

其中：
- 基础时间：操作本身耗时（如Kubeadm init约2分钟）
- 单节点时间：每个节点额外耗时（如镜像拉取约1分钟）
- 网络延迟补偿：离线环境增加50%，跨机房增加100%
```

### 4.5 重试终止条件

**[MUST]** 重试必须在以下条件之一满足时终止：

```go
func (c *Command) shouldStopRetry() bool {
    // [MUST] 达到最大重试次数
    if c.retryCount >= c.spec.BackoffLimit {
        return true
    }
    
    // [MUST] 达到执行超时
    if time.Since(c.startTime) > time.Duration(c.spec.ActiveDeadlineSecond)*time.Second {
        return true
    }
    
    // [MUST] 遇到不可重试错误
    if isUnrecoverableError(c.lastError) {
        return true
    }
    
    // [MUST] 资源被删除
    if c.isDeleted() {
        return true
    }
    
    return false
}
```

## 五、状态管理规范

### 5.1 节点状态管理规范

**[MUST]** 节点状态必须遵循状态机转换规则：

```
┌─────────────────────────────────────────────────────────────┐
│  节点状态转换图                                               │
└─────────────────────────────────────────────────────────────┘

Unknown ──→ Initializing ──→ BootStrapping ──→ Ready
    │            │                │              │
    │            │                │              ├──→ Upgrading ──→ Ready
    │            │                │              │        │
    │            │                │              │        └──→ UpgradeFailed
    │            │                │              │
    │            │                │              └──→ Deleting ──→ (Deleted)
    │            │                │
    │            │                └──→ BootStrapFailed
    │            │                        │
    │            │                        └──→ (NeedSkip=true)
    │            │
    │            └──→ InitFailed
    │                     │
    │                     └──→ (NeedSkip=true 或 人工介入)
    │
    └──→ (无有效转换)
```

**[MUST]** 状态转换必须满足：

```go
// 状态转换验证函数
func canTransitionTo(current, target NodeState) bool {
    // [MUST] 定义合法转换
    validTransitions := map[NodeState][]NodeState{
        NodeUnknown:    {NodeInitializing},
        NodeInitializing: {NodeInitFailed, NodeBootStrapping},
        NodeBootStrapping: {NodeBootStrapFailed, NodeReady},
        NodeReady:      {NodeUpgrading, NodeDeleting, NodeNotReady},
        NodeNotReady:   {NodeReady, NodeUpgrading, NodeDeleting},
        NodeUpgrading:  {NodeReady, NodeUpgradeFailed},
        NodeUpgradeFailed: {NodeUpgrading},  // 允许重试
        // [MUST] Failed状态只能转换到删除或重试
        NodeInitFailed:     {},
        NodeBootStrapFailed: {},
        NodeDeleteFailed:   {NodeDeleting},  // 允许重试删除
    }
    
    allowed, exists := validTransitions[current]
    if !exists {
        return false
    }
    
    for _, s := range allowed {
        if s == target {
            return true
        }
    }
    return false
}
```

### 5.2 状态持久化规范

**[MUST]** 状态变更必须立即持久化：

```go
// 标准状态更新模式
func (e *Phase) updateNodeState(nodeIP string, state NodeState, message string) error {
    // [MUST] 1. 检查状态转换合法性
    currentState := e.getNodeState(nodeIP)
    if !canTransitionTo(currentState, state) {
        return errors.Errorf("invalid state transition: %s -> %s", 
            currentState, state)
    }
    
    // [MUST] 2. 设置新状态
    e.setNodeState(nodeIP, state, message)
    
    // [MUST] 3. 立即持久化
    if err := mergecluster.SyncStatusUntilComplete(e.Client, e.BKECluster); err != nil {
        // [MUST] 4. 持久化失败必须返回错误
        return errors.Wrap(err, "failed to sync node state")
    }
    
    // [MUST] 5. 记录状态变更日志
    log.Info("Node state changed: %s -> %s, message: %s", 
        currentState, state, message)
    
    return nil
}
```

**[MUST]** 状态持久化失败处理：
1. 必须返回错误，触发Requeue
2. 必须记录Error级别日志
3. 不得继续执行后续操作

### 5.3 Phase状态管理规范

**[MUST]** Phase状态必须通过ExecutePreHook/ExecutePostHook自动管理：

```go
func (e *Phase) ExecutePreHook() error {
    // [MUST] 自动设置Phase为Running
    e.SetStatus(bkev1beta1.PhaseRunning)
    return e.BasePhase.DefaultPreHook()
}

func (e *Phase) ExecutePostHook(execErr error) error {
    // [MUST] 根据执行结果设置Phase状态
    if execErr == nil {
        e.SetStatus(bkev1beta1.PhaseSucceeded)
    } else {
        e.SetStatus(bkev1beta1.PhaseFailed)
    }
    return e.BasePhase.DefaultPostHook(execErr)
}
```

## 六、日志记录规范

### 6.1 日志级别使用规范

**[MUST]** 日志级别必须按照以下标准使用：

| 级别 | 使用场景 | 示例 |
|------|---------|------|
| **Info** | 正常流程关键步骤、状态变更、进度信息 | `log.Info("Master node init started")` |
| **Warn** | 可恢复异常、非预期情况、降级处理 | `log.Warn("Agent not ready, waiting...")` |
| **Error** | 严重错误、流程中断、需人工介入 | `log.Error("Master init failed: %v", err)` |
| **Debug** | 调试信息、详细执行过程（生产环境禁用） | `log.Debug("Poll count: %d", count)` |

**[MUST]** 日志输出频率控制：

```go
// [MUST] 轮询场景必须控制日志频率
const logInterval = 10  // 每10次轮询输出一次

for i := 0; i < maxPoll; i++ {
    // 执行轮询逻辑
    
    // [MUST] 使用取模控制频率
    if i % logInterval == 0 {
        log.Info("Polling status, count: %d", i)
    }
}
```

### 6.2 日志内容规范

**[MUST]** 日志必须包含以下要素：

```go
// 标准日志格式
log.<Level>(
    constant.<Reason>,           // [MUST] 事件Reason
    "<Message>",                 // [MUST] 人类可读的消息
    "<Key1>", value1,            // [MUST] 关键上下文信息
    "<Key2>", value2,
)

// 示例
log.Error(
    constant.MasterInitFailedReason,
    "Master initialization failed",
    "node", nodeIP,
    "error", err,
    "retryCount", retryCount,
)
```

**[MUST]** 日志内容要求：
1. 必须包含事件Reason（用于事件记录）
2. 必须包含人类可读的消息
3. 必须包含关键上下文（节点IP、资源名称等）
4. 不得包含敏感信息（密码、证书内容等）

### 6.3 事件Reason定义规范

**[MUST]** Reason命名必须遵循规范：

```
格式: <Resource><Operation><Status>

其中：
- Resource: 资源类型
- Operation: 操作类型
- Status: 结果状态

示例：
- MasterInitSucceed: Master初始化成功
- MasterInitFailed: Master初始化失败
- WorkerJoining: Worker加入中
- BKEAgentNotReady: Agent未就绪
```

**[MUST]** Reason必须在常量文件中统一定义：

```go
// 文件: utils/capbke/constant/constants.go
const (
    // Master相关Reason
    MasterInitReason           = "MasterInit"
    MasterNotInitReason        = "MasterNotInit"
    MasterInitFailedReason     = "MasterInitFailed"
    MasterInitSucceedReason    = "MasterInitSucceed"
    
    // [MUST] 每个Reason必须有注释说明
    // MasterInitFailedReason (Severity=Error) indicates master initialization failed
)
```

## 七、代码实现规范

### 7.1 标准Phase实现模板

**[MUST]** 所有Phase必须遵循以下实现模板：

```go
// 文件: pkg/phaseframe/phases/ensure_xxx.go

// [MUST] 1. 定义Phase名称常量
const (
    EnsureXxxName confv1beta1.BKEClusterPhase = "EnsureXxx"
    
    // [MUST] 定义超时、间隔等常量
    XxxTimeout = 10 * time.Minute
    XxxPollInterval = 2 * time.Second
)

// [MUST] 2. 定义Phase结构体
type EnsureXxx struct {
    phaseframe.BasePhase
    // [SHOULD] 添加Phase特定字段
}

// [MUST] 3. 实现构造函数
func NewEnsureXxx(ctx *phaseframe.PhaseContext) phaseframe.Phase {
    base := phaseframe.NewBasePhase(ctx, EnsureXxxName)
    return &EnsureXxx{BasePhase: base}
}

// [MUST] 4. 实现NeedExecute方法
func (e *EnsureXxx) NeedExecute(old, new *bkev1beta1.BKECluster) bool {
    // [MUST] 调用默认检查
    if !e.BasePhase.DefaultNeedExecute(old, new) {
        return false
    }
    
    // [MUST] 添加Phase特定检查
    needExecute := e.checkSpecificCondition()
    
    // [MUST] 设置Phase状态
    if needExecute {
        e.SetStatus(bkev1beta1.PhaseWaiting)
    }
    
    return needExecute
}

// [MUST] 5. 实现Execute方法
func (e *EnsureXxx) Execute() (ctrl.Result, error) {
    ctx, c, bkeCluster, _, log := e.Ctx.Untie()
    
    // [MUST] 5.1 前置检查
    if !e.preCheck() {
        log.Info("precondition not met")
        return ctrl.Result{RequeueAfter: quickRequeueInterval}, nil
    }
    
    // [MUST] 5.2 获取资源
    resources, err := e.getResources()
    if err != nil {
        // [MUST] 区分错误类型
        if apierrors.IsNotFound(err) {
            return ctrl.Result{}, nil
        }
        log.Error("failed to get resources: %v", err)
        return ctrl.Result{}, err
    }
    
    // [MUST] 5.3 执行核心逻辑（根据策略选择模板）
    result, err := e.doExecute(resources)
    if err != nil {
        return result, err
    }
    
    // [MUST] 5.4 后置处理
    if err := e.postProcess(); err != nil {
        log.Warn("post process failed: %v", err)
    }
    
    return ctrl.Result{}, nil
}

// [MUST] 6. 实现PreHook/PostHook（可选）
func (e *EnsureXxx) ExecutePreHook() error {
    return e.BasePhase.DefaultPreHook()
}
```
