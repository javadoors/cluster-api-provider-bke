## 一、异常分类体系

### 1.1 按严重程度分类

| 分类 | 说明 | 处理策略 | 示例 |
|------|------|---------|------|
| **致命错误** | 导致当前操作无法继续，需立即中断 | Fail-Fast，返回错误，等待人工介入 | Master升级失败、Etcd集群不可用 |
| **可恢复错误** | 临时性错误，可通过重试或等待恢复 | Requeue或RequeueAfter，自动重试 | Agent未就绪、网络超时、资源暂时不可用 |
| **警告错误** | 不影响主流程，但需要记录 | 记录日志和事件，继续执行 | 部分节点跳过、配置项缺失使用默认值 |
| **业务异常** | 业务逻辑层面的异常情况 | 标记状态，触发相应处理流程 | 节点NotReady、集群Unhealthy |

### 1.2 按来源分类

| 来源 | 说明 | 典型场景 |
|------|------|---------|
| **基础设施异常** | K8s API、网络、存储等底层异常 | API Server不可达、ConfigMap不存在、Secret访问失败 |
| **节点异常** | 节点层面的异常 | SSH连接失败、Agent未就绪、节点NotReady |
| **集群异常** | 集群层面的异常 | Etcd不健康、控制平面不可用、版本不兼容 |
| **配置异常** | 配置错误或缺失 | BKEConfig无效、版本不支持、参数校验失败 |
| **命令执行异常** | Agent命令执行失败 | Kubeadm执行失败、脚本执行错误、超时 |

## 二、异常处理策略矩阵

### 2.1 Phase层异常处理策略

```
┌─────────────────────────────────────────────────────────────┐
│  Phase层异常处理决策树                                       │
└─────────────────────────────────────────────────────────────┘
         │
         ├── 1. NeedExecute阶段异常
         │   ├── 资源不存在 → 返回false，跳过执行
         │   ├── 状态不满足 → 返回false，等待下次协调
         │   └── 配置错误 → 返回false，记录错误日志
         │
         ├── 2. Execute阶段异常
         │   │
         │   ├── 2.1 前置检查失败
         │   │   ├── 前置条件不满足 → RequeueAfter等待
         │   │   └── 前置检查错误 → 返回错误，中断流程
         │   │
         │   ├── 2.2 核心逻辑执行失败
         │   │   │
         │   │   ├── Master/Etcd操作 → Fail-Fast策略
         │   │   │   ├── 标记节点状态为Failed
         │   │   │   ├── 记录错误日志和事件
         │   │   │   ├── 更新Phase状态为Failed
         │   │   │   └── 立即返回错误，中断整个升级流程
         │   │   │
         │   │   └── Worker操作 → Best-Effort策略
         │   │       ├── 标记当前节点Failed
         │   │       ├── 记录失败节点列表
         │   │       ├── 继续处理下一个节点
         │   │       └── 全部处理完后，如有失败节点则返回错误
         │   │
         │   ├── 2.3 后置处理失败
         │   │   ├── 状态更新失败 → 记录错误，Requeue重试
         │   │   └── 清理工作失败 → 记录警告，不影响主流程
         │   │
         │   └── 2.4 等待超时
         │       ├── 命令执行超时 → 标记Failed，根据策略决定是否重试
         │       └── 健康检查超时 → 标记Failed，RequeueAfter重试
         │
         └── 3. ExecutePostHook阶段异常
             ├── 状态持久化失败 → 记录错误，返回错误
             └── 后置Hook失败 → 记录错误，不影响Phase结果
```

### 2.2 Controller层异常处理策略

```go
// 标准Reconcile异常处理模式
func (r *Reconciler) Reconcile(ctx context.Context, req ctrl.Request) (ctrl.Result, error) {
    // 1. 获取资源
    obj, err := r.get(ctx, req)
    if err != nil {
        if apierrors.IsNotFound(err) {
            // 资源不存在，正常结束
            return ctrl.Result{}, nil
        }
        // 其他错误，记录并返回
        return ctrl.Result{}, err
    }
    
    // 2. 前置检查
    if !r.shouldReconcile(obj) {
        // 不需要协调，正常结束
        return ctrl.Result{}, nil
    }
    
    // 3. 核心逻辑
    result, err := r.doReconcile(ctx, obj)
    if err != nil {
        // 根据错误类型决定处理策略
        if isRecoverable(err) {
            // 可恢复错误，RequeueAfter重试
            return ctrl.Result{RequeueAfter: 10 * time.Second}, nil
        }
        // 不可恢复错误，返回错误
        return ctrl.Result{}, err
    }
    
    return result, nil
}
```

## 三、状态管理规范

### 3.1 节点状态管理

**状态定义** (参考 `api/capbke/v1beta1/bkecluster_consts.go`):

```go
const (
    // 初始化阶段
    NodeUnknown       NodeState = "Unknown"        // 未知状态
    NodeInitializing  NodeState = "Initializing"   // 初始化中
    NodeInitFailed    NodeState = "InitFailed"     // 初始化失败
    
    // 引导阶段
    NodeBootStrapping   NodeState = "BootStrapping"   // 引导中
    NodeBootStrapFailed NodeState = "BootStrapFailed" // 引导失败
    
    // 运行阶段
    NodeReady    NodeState = "Ready"    // 就绪
    NodeNotReady NodeState = "NotReady" // 未就绪
    
    // 升级阶段
    NodeUpgrading     NodeState = "Upgrading"     // 升级中
    NodeUpgradeFailed NodeState = "UpgradeFailed" // 升级失败
    
    // 删除阶段
    NodeDeleting     NodeState = "Deleting"     // 删除中
    NodeDeleteFailed NodeState = "DeleteFailed" // 删除失败
    
    // 纳管阶段
    NodeManaging     NodeState = "Managing"     // 纳管中
    NodeManageFailed NodeState = "ManageFailed" // 纳管失败
)
```

**状态转换规则**:

```
正常流程:
  Unknown → Initializing → BootStrapping → Ready → (Upgrading → Ready)* → Deleting

异常流程:
  Initializing → InitFailed → (人工介入或自动重试)
  BootStrapping → BootStrapFailed → (标记NeedSkip或重试)
  Upgrading → UpgradeFailed → (Fail-Fast或Best-Effort)
  Deleting → DeleteFailed → (人工介入)
```

**状态管理代码示例**:

```go
// 标准状态设置模式
func (e *Phase) upgradeNode(node Node) error {
    ctx, c, bkeCluster, _, log := e.Ctx.Untie()
    nodeFetcher := e.Ctx.NodeFetcher()
    
    // 1. 标记开始状态
    nodeFetcher.SetNodeStateWithMessageForCluster(ctx, bkeCluster, node.IP, 
        bkev1beta1.NodeUpgrading, "Upgrading")
    if err := mergecluster.SyncStatusUntilComplete(c, bkeCluster); err != nil {
        return err  // 状态持久化失败，立即返回
    }
    
    // 2. 执行核心逻辑
    err := e.doUpgrade(node)
    
    // 3. 根据结果设置最终状态
    if err != nil {
        log.Error(constant.MasterUpgradeFailedReason, 
            "upgrade node %q failed: %v", node.IP, err)
        nodeFetcher.SetNodeStateWithMessageForCluster(ctx, bkeCluster, node.IP,
            bkev1beta1.NodeUpgradeFailed, err.Error())
    } else {
        nodeFetcher.SetNodeStateWithMessageForCluster(ctx, bkeCluster, node.IP,
            bkev1beta1.NodeNotReady, "Upgrading success")
    }
    
    // 4. 持久化状态
    if syncErr := mergecluster.SyncStatusUntilComplete(c, bkeCluster); syncErr != nil {
        return syncErr  // 状态持久化失败优先返回
    }
    
    return err  // 返回原始错误
}
```

## 四、重试和Requeue机制

### 4.1 Requeue策略分类

| 策略 | 使用场景 | 代码示例 | 说明 |
|------|---------|---------|------|
| **立即Requeue** | 需要立即重新协调 | `return ctrl.Result{Requeue: true}, nil` | 不等待，立即重新入队 |
| **延迟Requeue** | 需要等待一段时间后重试 | `return ctrl.Result{RequeueAfter: 10*time.Second}, nil` | 等待指定时间后重新入队 |
| **正常返回** | 操作成功，无需重试 | `return ctrl.Result{}, nil` | 不重新入队，等待下次事件触发 |
| **错误返回** | 操作失败，需要重试 | `return ctrl.Result{}, err` | Controller会自动Requeue |

### 4.2 Requeue时间常量定义

```go
const (
    // 快速重试间隔（用于临时性错误）
    quickRequeueInterval  = 10 * time.Second
    
    // 定期检查间隔（用于周期性健康检查）
    periodicCheckInterval = 5 * time.Minute
    
    // 节点操作轮询间隔
    nodePollInterval      = 2 * time.Second
    
    // 健康检查超时
    healthCheckTimeout    = 5 * time.Minute
    
    // 命令执行超时
    commandTimeout        = 10 * time.Minute
)
```

### 4.3 Requeue决策树

```
异常发生
│
├── 资源不存在
│   └── return ctrl.Result{}, nil  // 正常结束，不Requeue
│
├── 临时性错误（网络超时、资源暂时不可用）
│   └── return ctrl.Result{RequeueAfter: quickRequeueInterval}, nil
│
├── 前置条件不满足（Agent未就绪、节点NotReady）
│   └── return ctrl.Result{RequeueAfter: quickRequeueInterval}, nil
│
├── 配置错误（版本不支持、参数无效）
│   └── return ctrl.Result{}, err  // 返回错误，记录日志
│
├── 核心逻辑失败（Master升级失败、Etcd不可用）
│   ├── Master/Etcd → return ctrl.Result{}, err  // Fail-Fast
│   └── Worker → 记录失败，继续下一个
│
└── 状态持久化失败
    └── return ctrl.Result{}, err  // 必须重试
```

### 4.4 命令重试机制

**Command CRD重试配置**:

```go
type CommandSpec struct {
    // 最大重试次数（默认0，不重试）
    BackoffLimit int `json:"backoffLimit,omitempty"`
    
    // 重试间隔（默认0秒）
    BackoffDelay int `json:"backoffDelay,omitempty"`
    
    // 执行超时（默认600秒）
    ActiveDeadlineSecond int `json:"activeDeadlineSecond,omitempty"`
    
    // 失败是否跳过（默认false）
    BackoffIgnore bool `json:"backoffIgnore,omitempty"`
}
```

**重试逻辑示例**:

```go
// 命令执行失败处理
if err != nil {
    condition.Count++  // 增加重试计数
    
    if condition.Count >= command.Spec.BackoffLimit {
        // 达到最大重试次数
        if command.Spec.BackoffIgnore {
            // 标记为跳过，继续执行下一个命令
            condition.Phase = CommandSkip
            condition.Status = metav1.ConditionTrue
        } else {
            // 标记为失败，中断流程
            condition.Phase = CommandFailed
            condition.Status = metav1.ConditionFalse
        }
    } else {
        // 未达到最大重试次数，等待重试
        condition.Phase = CommandPending
        time.Sleep(time.Duration(command.Spec.BackoffDelay) * time.Second)
    }
}
```

## 七、代码示例和最佳实践

### 7.1 标准Phase异常处理模板

```go
func (e *EnsureSomePhase) Execute() (ctrl.Result, error) {
    ctx, c, bkeCluster, _, log := e.Ctx.Untie()
    
    // ========== 1. 前置检查 ==========
    if !e.preCheck() {
        log.Info(constant.SomeReason, "precondition not met, waiting...")
        return ctrl.Result{RequeueAfter: quickRequeueInterval}, nil
    }
    
    // ========== 2. 获取资源 ==========
    resources, err := e.getResources()
    if err != nil {
        if apierrors.IsNotFound(err) {
            log.Info(constant.SomeReason, "resource not found, skip")
            return ctrl.Result{}, nil  // 正常结束
        }
        log.Error(constant.SomeFailedReason, "failed to get resources: %v", err)
        return ctrl.Result{}, err  // 返回错误，自动Requeue
    }
    
    // ========== 3. 核心逻辑 ==========
    for _, resource := range resources {
        // 3.1 状态标记
        e.markStart(resource)
        if err := mergecluster.SyncStatusUntilComplete(c, bkeCluster); err != nil {
            return ctrl.Result{}, err  // 状态持久化失败
        }
        
        // 3.2 执行操作
        err := e.doOperation(resource)
        
        // 3.3 结果处理
        if err != nil {
            e.markFailed(resource, err)
            if syncErr := mergecluster.SyncStatusUntilComplete(c, bkeCluster); syncErr != nil {
                return ctrl.Result{}, syncErr
            }
            
            // 根据策略决定是否继续
            if e.shouldFailFast() {
                return ctrl.Result{}, err  // Fail-Fast
            }
            // Best-Effort: 记录失败，继续下一个
            continue
        }
        
        // 3.4 成功标记
        e.markSuccess(resource)
        if err := mergecluster.SyncStatusUntilComplete(c, bkeCluster); err != nil {
            return ctrl.Result{}, err
        }
    }
    
    // ========== 4. 后置处理 ==========
    if err := e.postProcess(); err != nil {
        log.Warn(constant.SomeWarningReason, "post process failed: %v", err)
        // 后置处理失败不影响主流程
    }
    
    log.Info(constant.SomeSucceedReason, "phase completed successfully")
    return ctrl.Result{}, nil
}
```

### 7.2 标准错误处理函数

```go
// 判断是否为可恢复错误
func isRecoverable(err error) bool {
    if apierrors.IsNotFound(err) {
        return false  // 资源不存在，不可恢复
    }
    if apierrors.IsConflict(err) {
        return true  // 版本冲突，可重试恢复
    }
    if errors.Is(err, context.DeadlineExceeded) {
        return true  // 超时，可重试恢复
    }
    if strings.Contains(err.Error(), "connection refused") {
        return true  // 网络错误，可重试恢复
    }
    return false
}

// 聚合多个错误
func aggregateErrors(errs ...error) error {
    var validErrs []error
    for _, err := range errs {
        if err != nil {
            validErrs = append(validErrs, err)
        }
    }
    if len(validErrs) == 0 {
        return nil
    }
    return kerrors.NewAggregate(validErrs)
}

// 标准错误处理包装器
func handleError(err error, operation string, log *bkev1beta1.BKELogger) (ctrl.Result, error) {
    if err == nil {
        return ctrl.Result{}, nil
    }
    
    log.Error(constant.InternalErrorReason, "%s failed: %v", operation, err)
    
    if isRecoverable(err) {
        return ctrl.Result{RequeueAfter: quickRequeueInterval}, nil
    }
    
    return ctrl.Result{}, err
}
```

### 7.3 状态转换最佳实践

```go
// 状态转换函数应具备幂等性
func (e *Phase) markNodeState(nodeIP string, state NodeState, message string) error {
    ctx, c, bkeCluster, _, _ := e.Ctx.Untie()
    nodeFetcher := e.Ctx.NodeFetcher()
    
    // 1. 检查当前状态（幂等性）
    currentState, _ := nodeFetcher.GetNodeStateForCluster(ctx, bkeCluster, nodeIP)
    if currentState == state {
        return nil  // 已处于目标状态，无需更新
    }
    
    // 2. 设置新状态
    nodeFetcher.SetNodeStateWithMessageForCluster(ctx, bkeCluster, nodeIP, state, message)
    
    // 3. 持久化状态
    return mergecluster.SyncStatusUntilComplete(c, bkeCluster)
}

// 状态转换应遵循状态机规则
func (e *Phase) canTransitionTo(currentState, targetState NodeState) bool {
    transitions := map[NodeState][]NodeState{
        NodeReady:    {NodeUpgrading, NodeDeleting},
        NodeUpgrading: {NodeReady, NodeUpgradeFailed},
        NodeUpgradeFailed: {NodeUpgrading},  // 允许重试
        // ... 其他转换规则
    }
    
    allowedStates, ok := transitions[currentState]
    if !ok {
        return false
    }
    
    for _, s := range allowedStates {
        if s == targetState {
            return true
        }
    }
    return false
}
```
