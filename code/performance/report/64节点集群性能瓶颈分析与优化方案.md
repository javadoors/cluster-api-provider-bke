## 2. 瓶颈根因深度分析

### 2.1 瓶颈 1: API Throttling (P0)

#### 2.1.1 现象描述

**日志证据** (bke-cluster-create.log L4-60):

```
I0708 10:49:50.679394 ... Waited for 1.01982684s due to client-side throttling,
  request: GET:https://192.168.200.95:6443/apis/runtime.cluster.x-k8s.io/v1alpha1
I0708 10:50:00.877197 ... Waited for 2.39753992s due to client-side throttling,
  request: GET:https://192.168.200.95:6443/apis/resource.k8s.io/v1
...
I0708 10:59:23.643086 ... Waited for 6.5973529s due to client-side throttling,
  request: GET:https://192.168.200.95:6443/apis/node.k8s.io/v1
```

**统计数据**:
- Throttling 次数: **57 次**
- 持续时间: **9 分 37 秒** (10:49:50 - 10:59:23)
- 平均等待时间: **~10 秒/次**
- 最大等待时间: **9.20 秒**
- 目标 API Server: 管理集群 (192.168.200.95:6443)

#### 2.1.2 根因分析

**根本原因**: Kubernetes client-go 默认客户端限流配置过于保守

```go
// 默认配置 (client-go/rest/config.go)
QPS:   5      // 每秒请求数
Burst: 10     // 突发请求数
```

**触发机制**:
1. BKE 控制器启动时，需要对管理集群做全量 API 资源发现 (Discovery)
2. 管理集群注册了 **30+ 个 APIGroup** (cluster.x-k8s.io, bootstrap.cluster.x-k8s.io, bke.bocloud.com 等)
3. 每个 APIGroup 需要查询多个版本 (v1alpha3, v1alpha4, v1beta1 等)
4. 总请求数: 30+ APIGroup × 2-3 版本 = **60-90 个请求**
5. 以 QPS=5 的速率发送，需要 **12-18 秒**
6. 但由于 Burst=10，前 10 个请求立即发送，后续请求被限流
7. 限流退避算法导致等待时间递增: 1s → 2s → 3s → ... → 9s

**影响链路**:

```
BKE 控制器启动
  ↓
初始化 Kubernetes client
  ↓
API 资源发现 (Discovery)
  ↓
遍历查询所有 APIGroup (60-90 个请求)
  ↓
触发 client-go 限流 (QPS=5, Burst=10)
  ↓
等待时间递增 (1s → 9s)
  ↓
总耗时 9 分 37 秒
```

#### 2.1.3 影响评估

| 维度 | 影响 |
|------|------|
| **时间成本** | 占总耗时 32.6%，是最大瓶颈 |
| **资源浪费** | 纯等待，无实际部署操作 |
| **用户体验** | 集群创建前 10 分钟无任何进展 |
| **优化潜力** | 投入产出比最高，改动最小 |

### 2.2 瓶颈 2: 健康检查收敛慢 (P1)

#### 2.2.1 现象描述

**日志证据** (bke-cluster-create.log L604-612):

```
11:12:04 [Warning] ClusterUnhealthy: no pods with prefix 'metrics-server-' in kube-system,
  pod ingress-nginx/ingress-nginx-controller-2zpcw unhealthy: status: Pending,
  pod ingress-nginx/ingress-nginx-controller-495xv unhealthy: status: Pending,
  ... (64 个 ingress-nginx Pod Pending)

11:15:08 [Warning] ClusterUnhealthy: node m1 NotReady
  calico-node-n2zqv, etcd-m1, kube-apiserver-m1, kube-controller-manager-m1 异常

11:16:11 [Warning] ClusterUnhealthy: node m2 NotReady
  calico-node-4phm7, etcd-m2, kube-apiserver-m2 异常

11:17:19 [Warning] ClusterUnhealthy: node m3 NotReady
  calico-node-wbxhf, etcd-m3, kube-apiserver-m3 异常

11:19:18 [Info] ClusterReady: 全部就绪
```

**统计数据**:
- ClusterUnhealthy 次数: **33 次**
- 持续时间: **7 分 14 秒** (11:12:04 - 11:19:18)
- Master NotReady 次数: **3 次** (m1, m2, m3 各 1 次)
- 关键阻塞组件: metrics-server, openfuyao-system-controller

#### 2.2.2 根因分析

**问题 1: Master 节点反复 NotReady**

**时序分析**:

```
11:15:08  m1 NotReady  (calico-node, etcd, kube-apiserver 等异常)
11:16:11  m2 NotReady  (calico-node, etcd, kube-apiserver 等异常)
11:17:19  m3 NotReady  (calico-node, etcd, kube-apiserver 等异常)
```

**模式特征**:
- 三个 Master 依次异常，间隔约 **1 分钟**
- 异常组件高度一致: calico-node, etcd, kube-apiserver, kube-controller-manager
- 每次异常持续约 **30-60 秒** 后自动恢复

**可能原因**:

1. **Calico 网络配置变更触发组件重启**
   - Calico Addon 部署时间: 11:07:33 - 11:10:48 (3 分 15 秒)
   - Master NotReady 时间: 11:15:08 - 11:17:19
   - 时间差: 4-7 分钟
   - 推测: Calico 网络初始化完成后，触发了控制面组件的网络重配置

2. **资源竞争导致组件 OOM/Restart**
   - 64 节点同时拉取镜像和初始化
   - Master 节点资源紧张 (CPU/内存)
   - 控制面组件被 OOM Kill 或 Restart

3. **etcd 集群重新选举**
   - Calico 网络变更导致 etcd 节点间通信中断
   - etcd 触发重新选举
   - 选举期间 kube-apiserver 不可用

**因果关系确认**:

通过日志分析，确认 **Calico 未部署时，Master 节点的状态是 Ready**，Master NotReady 是由 Calico 部署引起的。

**证据分析**:

从日志中可以清晰地看到以下时间线：

```
11:06:01  Master 节点全部加入成功（MasterJoinSucceed）
          ↓
11:06:03  Worker 节点开始加入
          ↓
11:07:31  Worker 节点全部加入成功
          ↓
11:07:33  Calico 部署开始
          ↓
11:10:48  Calico 部署成功
          ↓
11:12:04  健康检查开始
          ↓
11:12:04 - 11:14:53  健康检查报告 Pod 状态问题
                      （metrics-server 不存在、ingress-nginx Pending 等）
                      ❌ 没有报告 Master NotReady
          ↓
11:15:08  ✅ 第一次报告 Master NotReady（m1）
          ↓
11:16:04  ✅ 第二次报告 Master NotReady（m2）
          ↓
11:17:19  ✅ 第三次报告 Master NotReady（m3）
          ↓
11:19:18  集群就绪（ClusterReady）
```

**关键发现**:

1. **Calico 未部署时，Master 节点是 Ready 的**
   - 11:06:01 Master 节点全部加入成功
   - 11:12:04 - 11:14:53 健康检查中**没有报告 Master NotReady**
   - 这说明 **11:06:01 - 11:15:08 期间（约 9 分钟），Master 节点一直是 Ready 的**

2. **Master NotReady 发生在 Calico 部署完成后**
   - Calico 部署完成：11:10:48
   - 第一次 Master NotReady：11:15:08（**延迟 4 分 20 秒**）
   - 第二次 Master NotReady：11:16:04（**延迟 5 分 16 秒**）
   - 第三次 Master NotReady：11:17:19（**延迟 6 分 31 秒**）

3. **Master NotReady 时，calico-node Pod 也不健康**
   - 11:15:08 m1 NotReady 时，`calico-node-n2zqv unhealthy`
   - 11:16:04 m2 NotReady 时，`calico-node-4phm7 unhealthy`
   - 11:17:19 m3 NotReady 时，`calico-node-wbxhf unhealthy`

**因果关系确认表**:

| 时间点 | Master 状态 | Calico 状态 | 说明 |
|--------|------------|------------|------|
| 11:06:01 | **Ready** | 未部署 | Master 加入成功 |
| 11:10:48 | **Ready** | 部署完成 | Calico 部署成功 |
| 11:12:04 - 11:14:53 | **Ready** | 运行中 | 健康检查未报告 Master NotReady |
| 11:15:08 | **NotReady** | 运行中 | 第一次 Master NotReady |
| 11:16:04 | **NotReady** | 运行中 | 第二次 Master NotReady |
| 11:17:19 | **NotReady** | 运行中 | 第三次 Master NotReady |

**结论**:
- ✅ **Calico 未部署时，Master 节点的状态是 Ready**
- ✅ **Master NotReady 是由 Calico 部署引起的**
- ✅ **因果关系成立**

**为什么延迟 4-7 分钟？**

可能的原因：
1. **Calico BGP 会话建立需要时间**：64 个节点需要建立 BGP 会话，可能需要几分钟
2. **Calico 网络策略应用需要时间**：网络策略需要逐步应用到所有节点
3. **资源竞争导致延迟**：64 个 calico-node Pod 同时启动，消耗大量 CPU/内存，导致 Master 组件响应变慢

这进一步证实了之前的优化方案（渐进式部署 Calico）是正确的方向。

**问题 2: 关键组件长时间 Pending**

**openfuyao-system-controller**:
- Manifest apply 时间: 11:11:40 (<1 秒)
- 首次健康检查: 11:12:04 (Pending)
- Pod 重建: 11:17:52 (7pm8v → zfkgv)
- 最终就绪: ~11:19:18
- Pending 总时长: **~7 分钟**

**metrics-server**:
- 直到 11:18:51 仍报告 "no pods with prefix 'metrics-server-'"
- 最终就绪: 11:19:18
- Pending 总时长: **~7 分钟**

**可能原因**:

1. **镜像拉取慢**
   - 64 节点同时拉取镜像
   - Registry 带宽瓶颈
   - 镜像未预置在节点上

2. **调度延迟**
   - 大量 Pod 同时创建
   - Scheduler 队列积压
   - 节点资源不足导致Pending

3. **依赖组件未就绪**
   - metrics-server 依赖 kubelet 汇报指标
   - kubelet 本身可能未完全就绪
   - 形成循环依赖

##### 维度 1: 确认因果关系

**目标**：确认 Master NotReady 是否由 Calico 部署引起

**1.1 时间线对比分析**

已有数据：
```
11:07:33  Calico 部署开始
11:10:48  Calico 部署成功（耗时 3 分 15 秒）
11:15:08  m1 NotReady（延迟 4 分 20 秒）
11:16:11  m2 NotReady（延迟 5 分 23 秒）
11:17:19  m3 NotReady（延迟 6 分 31 秒）
```

排查步骤：

```bash
# 1. 检查 Calico 部署完成后的事件
kubectl get events --sort-by='.lastTimestamp' -A | grep -E "calico|apiserver|etcd"

# 2. 检查 Master 节点状态变化时间
kubectl get nodes -o wide --watch

# 3. 检查 kube-apiserver Pod 重启时间
kubectl get pods -n kube-system -l component=kube-apiserver -o wide
```

关键问题：
- Master NotReady 是否发生在 Calico 部署完成后？✅ 是（延迟 4-7 分钟）
- 是否有其他操作同时发生？需要检查

**1.2 排除其他因素**

```bash
# 1. 检查是否有其他 Addon 同时部署
kubectl get events -A | grep -E "AddonDeploy" | grep "11:10:48\|11:11\|11:12\|11:13\|11:14"

# 2. 检查是否有节点资源不足
kubectl top nodes -l node-role.kubernetes.io/master

# 3. 检查是否有其他 Pod 大量创建
kubectl get pods -A --sort-by='.metadata.creationTimestamp' | tail -50
```

判断标准：
- 如果 Master NotReady 时间与 Calico 部署完成时间高度相关 → 因果关系成立
- 如果有其他操作同时发生 → 需要进一步分析

##### 维度 3: 分析根因机制

**目标**：确定具体的技术原因

**3.1 假设 1: BGP 会话建立导致 API Server 负载激增**

排查步骤：

```bash
# 1. 检查 API Server 请求延迟
kubectl get --raw /metrics | grep apiserver_request_latencies

# 2. 检查 API Server 请求数
kubectl get --raw /metrics | grep apiserver_request_count

# 3. 检查 etcd 请求延迟
kubectl exec -n kube-system etcd-m1 -- etcdctl endpoint status --write-out=table

# 4. 检查 BGP 会话数量
kubectl exec -n kube-system $(kubectl get pods -n kube-system -l k8s-app=calico-node -o name | head -1) -- calicoctl get bgppeer
```

判断标准：
- API Server 请求延迟是否显著增加？（> 1 秒）
- API Server 请求数是否激增？（> 1000 QPS）
- etcd 请求延迟是否增加？（> 100ms）
- BGP 会话数量是否为 N²（64² = 4096）？

**3.2 假设 2: 网络策略误拦截控制面通信**

排查步骤：

```bash
# 1. 检查控制面 Pod 的网络连通性
kubectl exec -n kube-system kube-apiserver-m1 -- curl -k https://localhost:6443/healthz

# 2. 检查 etcd 与 API Server 的连通性
kubectl exec -n kube-system kube-apiserver-m1 -- curl -k https://etcd-m1:2379/health

# 3. 检查 calico-node 的网络策略日志
kubectl logs -n kube-system -l k8s-app=calico-node | grep -E "policy|deny|drop"

# 4. 临时禁用 Calico 网络策略
kubectl patch globalnetworkpolicy default-deny -p '{"spec":{"ingress":[{"action":"Allow"}],"egress":[{"action":"Allow"}]}}'
```

判断标准：
- 控制面 Pod 之间是否无法通信？
- 禁用网络策略后是否恢复正常？
- calico-node 日志中是否有 deny/drop 记录？

**3.3 假设 3: 资源竞争导致控制面组件 OOM/Restart**

排查步骤：


### 2.3 瓶颈 3: Calico 部署慢 (P2)

#### 2.3.1 现象描述

**日志证据** (bke-cluster-create.log L449-450):

```
11:07:33 [Info] AddonDeploying: start to create addon "calico"
11:10:48 [Info] AddonDeploySucceeded: create calico/v3.31.3 success
```

**统计数据**:
- 部署耗时: **3 分 15 秒**
- 占 Addon 阶段比例: **78.6%** (5 个 Addon 总耗时 4 分 8 秒)
- 部署方式: DaemonSet (64 个节点)

#### 2.3.2 根因分析

**Calico 部署流程**:

```
1. 创建 Calico CRD 和 RBAC
2. 部署 calico-kube-controllers (Deployment)
3. 部署 calico-node (DaemonSet, 64 个节点)
4. 每个节点:
   ├─ 拉取 calico-node 镜像
   ├─ 启动 calico-node Pod
   ├─ 配置 BGP/IPIP/VXLAN 网络
   ├─ 初始化路由表
   └─ 注册到 Calico 控制面
```

**耗时环节**:

1. **镜像拉取** (~1 分钟)
   - calico-node 镜像大小: ~150MB
   - 64 节点同时拉取
   - Registry 带宽: 假设 100Mbps
   - 单节点拉取时间: 150MB × 8 / 100Mbps = 12 秒
   - 64 节点并发: 12 秒 × 64 / 并发数 = ~1 分钟

2. **网络初始化** (~1.5 分钟)
   - BGP 邻居建立: 每个节点需要与所有 Master 建立 BGP 会话
   - IPIP/VXLAN 隧道创建: 每个节点需要创建隧道接口
   - 路由表同步: 每个节点需要同步所有 Pod CIDR 路由
   - 64 节点 × 3 Master = 192 个 BGP 会话
   - 网络收敛时间: ~1.5 分钟

3. **控制面注册** (~30 秒)
   - 64 个 calico-node 向 calico-kube-controllers 注册
   - 控制面处理队列积压
   - 注册确认时间: ~30 秒

#### 2.3.3 影响评估

| 维度 | 影响 |
|------|------|
| **时间成本** | 占总耗时 14.0%，是第三大瓶颈 |
| **网络依赖** | 依赖 Registry 带宽和网络质量 |
| **优化潜力** | 中等，可通过镜像预置优化 |

```go
package kube

import (
    "context"
    
    "k8s.io/client-go/dynamic"
    "k8s.io/client-go/kubernetes"
    "k8s.io/client-go/rest"
    ctrl "sigs.k8s.io/controller-runtime"
    
    "gopkg.openfuyao.cn/cluster-api-provider-bke/utils/capbke/config"
)

// ApplyThrottlingConfig applies QPS/Burst throttling configuration to rest.Config
// This is the single source of truth for client throttling settings
func ApplyThrottlingConfig(cfg *rest.Config) *rest.Config {
    if cfg == nil {
        return cfg
    }
    
    // Apply QPS and Burst from centralized config
    cfg.QPS = config.ClientQPS
    cfg.Burst = config.ClientBurst
    
    return cfg
}

// NewClientFromConfig creates a new Kubernetes client with throttling applied
func NewClientFromConfig(cfg *rest.Config) (*kubernetes.Clientset, error) {
    cfg = ApplyThrottlingConfig(cfg)
    return kubernetes.NewForConfig(cfg)
}

// NewDynamicClientFromConfig creates a new dynamic client with throttling applied
func NewDynamicClientFromConfig(cfg *rest.Config) (dynamic.Interface, error) {
    cfg = ApplyThrottlingConfig(cfg)
    return dynamic.NewForConfig(cfg)
}

// GetManagerConfig returns a rest.Config for controller-runtime manager with throttling applied
func GetManagerConfig() *rest.Config {
    return ApplyThrottlingConfig(ctrl.GetConfigOrDie())
}

// NewRemoteKubeClient creates a RemoteKubeClient with throttling applied
func NewRemoteKubeClient(ctx context.Context, cfg *rest.Config) (RemoteKubeClient, error) {
    return NewClientFromRestConfig(ctx, ApplyThrottlingConfig(cfg))
}
```

##### 3. 方案 B: RESTMapper 缓存详细设计

###### 3.1 创建全局 RESTMapper 单例

**新增文件**: `pkg/kube/restmapper.go`

```go
package kube

import (
    "sync"
    
    "k8s.io/apimachinery/pkg/api/meta"
    "k8s.io/client-go/discovery"
    "k8s.io/client-go/discovery/cached/memory"
    "k8s.io/client-go/rest"
    "k8s.io/client-go/restmapper"
)

var (
    globalRESTMapper     meta.RESTMapper
    globalRESTMapperOnce sync.Once
    globalRESTMapperErr  error
)

// GetGlobalRESTMapper returns a global shared RESTMapper with caching.
//
// The RESTMapper is initialized once and cached for the lifetime of the process.
// This is safe because:
// - BKE CRDs are installed before the controller starts (via Helm/kubectl)
// - CRDs do not change during controller runtime
// - RESTMapper cache is stable for the lifetime of the process
//
// If CRDs need to be dynamically installed/deleted during runtime in the future,
// a cache invalidation mechanism can be added at that time.
func GetGlobalRESTMapper(config *rest.Config) (meta.RESTMapper, error) {
    globalRESTMapperOnce.Do(func() {
        discoveryClient, err := discovery.NewDiscoveryClientForConfig(config)
        if err != nil {
            globalRESTMapperErr = err
            return
        }
        
        // 使用内存缓存
        cachedDiscovery := memory.NewMemCacheClient(discoveryClient)
        
        // 创建延迟加载的 RESTMapper
        globalRESTMapper = restmapper.NewDeferredDiscoveryRESTMapper(cachedDiscovery)
    })
    
    return globalRESTMapper, globalRESTMapperErr
}
```

| 问题 | 描述 | 影响 |
|------|------|------|

#### 3.2.4 推荐方案: 统一优化方案

```
┌─────────────────────────────────────────────────────────────┐
│                    统一健康检查架构                            │
├─────────────────────────────────────────────────────────────┤
```

##### 核心设计原则

| 原则 | 说明 | 对应优化点 |
|------|------|-----------|
| **渐进式** | 按优先级分阶段检查，关键组件失败立即返回 | 渐进式检查 + 优先级检查 |
| **并行化** | 每个阶段内使用并行检查 | 并行检查 |
| **缓存化** | 使用缓存减少 API 调用 | 缓存机制 |
| **智能化** | 根据检查结果动态调整间隔 | 动态间隔 |

```go
// HealthCheckConfig 健康检查配置
type HealthCheckConfig struct {
    // 缓存配置
    CacheTTL time.Duration `yaml:"cacheTTL"`
    
    // 检查间隔配置
    CriticalComponentInterval  time.Duration `yaml:"criticalComponentInterval"`  // 关键组件失败：5 秒
    ImportantComponentInterval time.Duration `yaml:"importantComponentInterval"` // 重要组件失败：15 秒
    OptionalComponentInterval  time.Duration `yaml:"optionalComponentInterval"`  // 非关键组件失败：30 秒
    NormalInterval             time.Duration `yaml:"normalInterval"`             // 正常：5 分钟
    
    // 组件清单配置
    CriticalComponents  []ComponentCheck `yaml:"criticalComponents"`
    ImportantComponents []ComponentCheck `yaml:"importantComponents"`
    OptionalComponents  []ComponentCheck `yaml:"optionalComponents"`
}

// HealthCheckResult 健康检查结果
type HealthCheckResult struct {
    NodeErrors                []error
    CriticalComponentErrors   []error
    ImportantComponentErrors  []error
    OptionalComponentErrors   []error
}

// UnifiedHealthChecker 统一健康检查器（独立结构体）
type UnifiedHealthChecker struct {
    kubeClient kubernetes.Interface  // 使用 k8s 原生接口
    log        *log.Logger
    cache      *HealthCheckCache
    config     HealthCheckConfig
}

// NewUnifiedHealthChecker 创建健康检查器
func NewUnifiedHealthChecker(kubeClient kubernetes.Interface, log *log.Logger, config HealthCheckConfig) *UnifiedHealthChecker {
    return &UnifiedHealthChecker{
        kubeClient: kubeClient,
        log:        log,
        cache:      NewHealthCheckCache(config.CacheTTL),
        config:     config,
    }
}

// DefaultHealthCheckConfig 默认配置
func DefaultHealthCheckConfig() HealthCheckConfig {
    return HealthCheckConfig{

        CriticalComponentInterval:  5 * time.Second,
        ImportantComponentInterval: 15 * time.Second,
        OptionalComponentInterval:  30 * time.Second,

        },
        ImportantComponents: []ComponentCheck{

        },
        OptionalComponents: []ComponentCheck{
            {Namespace: "kube-system", Prefixes: []string{"metrics-server-"}},

            {Namespace: "monitoring", Prefixes: []string{"alertmanager-main-", "prometheus-k8s-", "node-exporter-"}},
            {Namespace: "openfuyao-system", Prefixes: []string{"console-service-", "oauth-server-", "local-harbor-"}},
        },
    }

// LoadHealthCheckConfig 从配置文件加载配置
func LoadHealthCheckConfig(configPath string) HealthCheckConfig {
    data, err := os.ReadFile(configPath)
    if err != nil {
        log.Warnf("failed to load health check config from %s, using default: %v", configPath, err)
        return DefaultHealthCheckConfig()
    }
    
    var config HealthCheckConfig
    if err := yaml.Unmarshal(data, &config); err != nil {
        log.Warnf("failed to parse health check config, using default: %v", err)
        return DefaultHealthCheckConfig()
    }
    
    // 如果某些字段为空，使用默认值
    defaultConfig := DefaultHealthCheckConfig()
    if len(config.CriticalComponents) == 0 {
        config.CriticalComponents = defaultConfig.CriticalComponents
    }
    if len(config.ImportantComponents) == 0 {
        config.ImportantComponents = defaultConfig.ImportantComponents
    }
    if len(config.OptionalComponents) == 0 {
        config.OptionalComponents = defaultConfig.OptionalComponents
    }
    
    return config
}

// CheckClusterHealth 统一健康检查入口（独立函数，不依赖具体 Client 实现）
func CheckClusterHealth(kubeClient kubernetes.Interface, log *log.Logger, cluster *bkev1beta1.BKECluster, currentVersion string, bkeNodes bkev1beta1.BKENodes) error {
    config := LoadHealthCheckConfig("/etc/bke/health-check-config.yaml")
    checker := NewUnifiedHealthChecker(kubeClient, log, config)
    return checker.Check(cluster, currentVersion, bkeNodes)
}

// Check 执行统一健康检查
func (h *UnifiedHealthChecker) Check(cluster *bkev1beta1.BKECluster, currentVersion string, bkeNodes bkev1beta1.BKENodes) error {
    result := &HealthCheckResult{}
    
    // 阶段 1: 节点状态检查（并行）
    if err := h.checkNodesParallel(cluster, currentVersion, bkeNodes, result); err != nil {
        result.NodeErrors = append(result.NodeErrors, err)
        return h.aggregateResult(result)
    }
    
    // 阶段 2: 关键组件检查（并行）
    if err := h.checkCriticalComponentsParallel(result); err != nil {
        result.CriticalComponentErrors = append(result.CriticalComponentErrors, err)
        return h.aggregateResult(result)
    }
    
    // 阶段 3: 重要组件检查（并行）
    if err := h.checkImportantComponentsParallel(result); err != nil {
        h.log.Warn("important components check failed: %v", err)
        result.ImportantComponentErrors = append(result.ImportantComponentErrors, err)
    }
    
    // 阶段 4: 非关键组件检查（并行）
    if err := h.checkOptionalComponentsParallel(result); err != nil {
        h.log.Debug("optional components check failed: %v", err)
        result.OptionalComponentErrors = append(result.OptionalComponentErrors, err)
    }
    
    return h.aggregateResult(result)
}

// checkNodesParallel 并行检查节点状态
func (h *UnifiedHealthChecker) checkNodesParallel(cluster *bkev1beta1.BKECluster, currentVersion string, bkeNodes bkev1beta1.BKENodes, result *HealthCheckResult) error {
    // 从缓存获取节点列表
    nodes, err := h.cache.GetNodes(h.kubeClient)
    if err != nil {
        return err
    }
    
    // 并行检查所有节点
    var wg sync.WaitGroup
    errChan := make(chan error, len(nodes.Items))
    
    for _, node := range nodes.Items {
        nodeIP := GetNodeIP(&node)
        
        // 跳过需要跳过的节点
        if bkeNodes.GetNodeStateNeedSkip(nodeIP) {
            continue
        }
        
        wg.Add(1)
        go func(n corev1.Node) {
            defer wg.Done()
            if err := h.checkNode(&n, currentVersion); err != nil {
                errChan <- err
            }
        }(node)
    }
    
    wg.Wait()
    close(errChan)
    
    // 收集错误
    for err := range errChan {
        result.NodeErrors = append(result.NodeErrors, err)
    }
    
    if len(result.NodeErrors) > 0 {
        return kerrors.NewAggregate(result.NodeErrors)
    }
    
    return nil
}

// checkCriticalComponentsParallel 并行检查关键组件
func (h *UnifiedHealthChecker) checkCriticalComponentsParallel(result *HealthCheckResult) error {
    return h.checkComponentsByPriority(h.config.CriticalComponents, result)
}

// checkImportantComponentsParallel 并行检查重要组件
func (h *UnifiedHealthChecker) checkImportantComponentsParallel(result *HealthCheckResult) error {
    return h.checkComponentsByPriority(h.config.ImportantComponents, result)
}

// checkOptionalComponentsParallel 并行检查非关键组件
func (h *UnifiedHealthChecker) checkOptionalComponentsParallel(result *HealthCheckResult) error {
    return h.checkComponentsByPriority(h.config.OptionalComponents, result)
}

// checkComponentsByPriority 按优先级并行检查组件
func (h *UnifiedHealthChecker) checkComponentsByPriority(components []ComponentCheck, result *HealthCheckResult) error {
    var wg sync.WaitGroup
    errChan := make(chan error, len(components))
    
    for _, check := range components {
        wg.Add(1)
        go func(c ComponentCheck) {
            defer wg.Done()
            if err := h.checkComponent(c); err != nil {
                errChan <- err
            }
        }(check)
    }
    
    wg.Wait()
    close(errChan)
    
    var errs []error
    for err := range errChan {
        errs = append(errs, err)
    }
    
    if len(errs) > 0 {
        return kerrors.NewAggregate(errs)
    }
    
    return nil
}

// aggregateResult 聚合检查结果
func (h *UnifiedHealthChecker) aggregateResult(result *HealthCheckResult) error {
    // 节点错误或关键组件错误，立即返回
    if len(result.NodeErrors) > 0 || len(result.CriticalComponentErrors) > 0 {
        var allErrors []error
        allErrors = append(allErrors, result.NodeErrors...)
        allErrors = append(allErrors, result.CriticalComponentErrors...)
        return kerrors.NewAggregate(allErrors)
    }
    
    // 重要组件错误，记录警告
    if len(result.ImportantComponentErrors) > 0 {
        h.log.Warn("important component errors: %v", result.ImportantComponentErrors)
    }
    
    // 非关键组件错误，记录调试信息
    if len(result.OptionalComponentErrors) > 0 {
        h.log.Debug("optional component errors: %v", result.OptionalComponentErrors)
    }
    
    h.log.Info("cluster health check pass")
    return nil
}

// GetRequeueInterval 根据检查结果动态调整间隔
func GetRequeueInterval(result *HealthCheckResult, config HealthCheckConfig) time.Duration {
    // 节点错误或关键组件错误，快速重试
    if len(result.NodeErrors) > 0 || len(result.CriticalComponentErrors) > 0 {
        return config.CriticalComponentInterval
    }
    
    // 重要组件错误，中速重试
    if len(result.ImportantComponentErrors) > 0 {
        return config.ImportantComponentInterval
    }
    
    // 非关键组件错误，慢速重试
    if len(result.OptionalComponentErrors) > 0 {
        return config.OptionalComponentInterval
    }
    
    // 正常，使用正常间隔
    return config.NormalInterval
}
```

```go
type HealthCheckCache struct {
    pods     map[string][]corev1.Pod
    nodes    *corev1.NodeList
    lastSync time.Time
    ttl      time.Duration
    mu       sync.RWMutex
}

// NewHealthCheckCache 创建缓存
func NewHealthCheckCache(ttl time.Duration) *HealthCheckCache {
    return &HealthCheckCache{
        pods: make(map[string][]corev1.Pod),
        ttl:  ttl,
    }
}

// GetNodes 从缓存获取节点列表
func (c *HealthCheckCache) GetNodes(client *Client) (*corev1.NodeList, error) {
    c.mu.RLock()
    if c.nodes != nil && time.Since(c.lastSync) < c.ttl {
        defer c.mu.RUnlock()
        return c.nodes, nil
    }
    c.mu.RUnlock()
    
    // 缓存过期，重新获取
    c.mu.Lock()
    defer c.mu.Unlock()
    
    nodes, err := client.ListNodes(nil)
    if err != nil {
        return nil, err
    }
    
    c.nodes = nodes
    c.lastSync = time.Now()
    
    return nodes, nil
}

// GetPods 从缓存获取 Pod 列表
func (c *HealthCheckCache) GetPods(client *Client, namespace string) ([]corev1.Pod, error) {
    c.mu.RLock()
    if pods, ok := c.pods[namespace]; ok && time.Since(c.lastSync) < c.ttl {
        defer c.mu.RUnlock()
        return pods, nil
    }
    c.mu.RUnlock()
    
    // 缓存过期，重新获取
    c.mu.Lock()
    defer c.mu.Unlock()
    
    pods, err := client.getPods(namespace)
    if err != nil {
        return nil, err
    }
    
    c.pods[namespace] = pods
    c.lastSync = time.Now()
    
    return pods, nil
}
```

#### 3.3.3 优化项 1: 镜像预置（节省 ~50 秒）

##### 原理

**当前问题**：
- Calico DaemonSet 创建后，64 个节点同时开始拉取镜像
- 每个节点需要拉取 4 个镜像：calico-node、calico-cni、calico-kube-controllers、calico-pod2daemon-flexvol
- 总镜像拉取量：64 节点 × 4 镜像 × ~100MB/镜像 = ~25.6GB
- 如果 Registry 带宽为 1Gbps，理论拉取时间：25.6GB × 8 / 1Gbps = ~200 秒

**优化思路**：
- 在节点环境初始化阶段（EnsureNodesEnv）提前拉取 Calico 镜像
- 此时集群尚未创建，网络压力较小，可以充分利用带宽
- DaemonSet 创建时，镜像已存在于节点本地，无需再次拉取

##### 影响分析

**正面影响**：
- 镜像拉取时间从 ~1 分钟降低到 ~10 秒
- 减少 DaemonSet 创建时的网络压力
- 提高部署的可预测性

**负面影响**：
- 节点环境初始化时间增加 ~10 秒（可接受）
- 如果预置失败，需要降级到正常拉取（已处理）

```go
// CalicoImages 定义需要预置的 Calico 镜像
var CalicoImages = []string{
    "registry.openfuyao.com/calico/node:v3.31.3",
    "registry.openfuyao.com/calico/cni:v3.31.3",
    "registry.openfuyao.com/calico/kube-controllers:v3.31.3",
    "registry.openfuyao.com/calico/pod2daemon-flexvol:v3.31.3",
}

// prePullCalicoImages 预置 Calico 镜像
func (e *EnsureNodesEnv) prePullCalicoImages(nodes bkenode.Nodes) error {
    var wg sync.WaitGroup
    errChan := make(chan error, len(nodes)*len(CalicoImages))
    
    // 并行预置：每个节点一个 goroutine
    for _, node := range nodes {
        wg.Add(1)
        go func(n bkenode.Node) {
            defer wg.Done()
            
            // 每个节点并行预置所有镜像
            var nodeWg sync.WaitGroup
            for _, image := range CalicoImages {
                nodeWg.Add(1)
                go func(img string) {
                    defer nodeWg.Done()
                    
                    // 检查镜像是否已存在
                    if e.imageExists(n, img) {
                        e.Ctx.Log.Debug("Calico image %s already exists on node %s", img, n.IP)
                        return
                    }
                    
                    // 预置镜像
                    cmd := fmt.Sprintf("crictl pull %s", img)
                    if err := e.executeOnNode(n, cmd); err != nil {
                        errChan <- fmt.Errorf("failed to pre-pull image %s on node %s: %v", img, n.IP, err)
                    } else {
                        e.Ctx.Log.Info("Successfully pre-pulled Calico image %s on node %s", img, n.IP)
                    }
                }(image)
            }
            nodeWg.Wait()
        }(node)
    }
    
    wg.Wait()
    close(errChan)
    
    // 收集错误
    var errs []error
    for err := range errChan {
        errs = append(errs, err)
    }
    
    if len(errs) > 0 {
        e.Ctx.Log.Warn("Some Calico images failed to pre-pull, will fallback to normal pull: %v", errs)
    }
    
    return nil
}

// imageExists 检查镜像是否已存在
func (e *EnsureNodesEnv) imageExists(node bkenode.Node, image string) bool {
    cmd := fmt.Sprintf("crictl images | grep -q '%s'", image)
    return e.executeOnNode(node, cmd) == nil
}
```

#### 3.3.4 优化项 2: 并行部署（节省 ~20 秒）

##### 原理

**当前问题**：
- DaemonSet 默认使用 RollingUpdate 策略，`maxUnavailable=1`
- 这意味着每次只有 1 个节点可以更新，其他节点需要等待
- 64 个节点需要 64 轮更新，每轮 ~1.5 秒，总计 ~96 秒

**优化思路**：
- 调整 DaemonSet 的 `maxUnavailable` 参数为 30%
- 允许 30% 的节点（~19 个节点）同时更新
- 减少更新轮数：64 / 19 = ~4 轮，每轮 ~1.5 秒，总计 ~6 秒

##### 影响分析

**正面影响**：
- 部署时间从 ~1.5 分钟降低到 ~1 分钟
- 提高部署效率

**负面影响**：
- 同时启动更多 Pod，增加资源压力（已通过资源限制缓解）
- 如果某个节点失败，影响范围更大（已通过 maxSurge=0 缓解）

##### 实现方案

```yaml
# calico-node DaemonSet 配置优化
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: calico-node
  namespace: kube-system
spec:
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 30%  # 允许 30% 的节点同时更新
      maxSurge: 0          # 不允许额外创建 Pod
  template:
    spec:
      # 启用并行拉取镜像
      imagePullSecrets:
      - name: registry-secret
      
      # 设置资源限制，避免资源竞争
      containers:
      - name: calico-node
        resources:
          requests:
            cpu: 100m
            memory: 128Mi
          limits:
            cpu: 500m
            memory: 512Mi
        
        # 启用 readinessProbe，加速 Pod 就绪
        readinessProbe:
          httpGet:
            path: /readiness
            port: 9080
          initialDelaySeconds: 0
          periodSeconds: 1
          timeoutSeconds: 1
          successThreshold: 1
          failureThreshold: 3
```

##### readinessProbe 加速 Pod 就绪原理

**Kubernetes Pod 就绪状态机制**：

Kubernetes 中 Pod 有三个关键状态：

| 状态 | 说明 | 影响 |
|------|------|------|
| **Pending** | Pod 已被接受，但容器尚未运行 | 不可用 |
| **Running** | 至少有一个容器正在运行 | 不一定可用 |
| **Ready** | Pod 可以接受流量 | **可用** |

**关键点**：`Running ≠ Ready`，Pod 必须通过 readinessProbe 才能进入 Ready 状态。

**readinessProbe vs livenessProbe**：

| Probe 类型 | 作用 | 失败后果 | 使用场景 |
|-----------|------|---------|---------|
| **readinessProbe** | 检查 Pod 是否可以接受流量 | 从 Service 中移除 Pod | 应用启动、依赖检查 |
| **livenessProbe** | 检查 Pod 是否存活 | 重启 Pod | 死锁检测、健康检查 |

**默认行为（无 readinessProbe 或配置不当）**：
- 如果没有 readinessProbe，Kubernetes 会在容器启动后立即将 Pod 标记为 Ready
- 但此时应用可能尚未完全初始化（如 Calico 尚未建立网络连接）
- 导致 Pod 虽然 Ready，但实际无法提供服务
- DaemonSet 控制器认为 Pod 已就绪，继续创建下一个 Pod，但实际上前一个 Pod 尚未真正就绪

**优化后的行为（配置 readinessProbe）**：
- 配置 readinessProbe，让 Kubernetes 等待应用真正就绪
- 只有当 readinessProbe 成功时，Pod 才被标记为 Ready
- DaemonSet 控制器看到 Pod Ready 后，继续创建下一个 Pod

**参数优化对比**：

| 参数 | 默认值 | 优化值 | 说明 |
|------|--------|--------|------|
| `initialDelaySeconds` | 0 | 0 | 容器启动后立即开始检查 |
| `periodSeconds` | 10 | **1** | 检查频率从 10 秒提高到 1 秒 |
| `timeoutSeconds` | 1 | 1 | 超时时间保持 1 秒 |
| `successThreshold` | 1 | 1 | 连续 1 次成功即就绪 |
| `failureThreshold` | 3 | 3 | 连续 3 次失败才认为未就绪 |

**为什么可以加速**：
- **默认配置**：每 10 秒检查一次，即使应用 1 秒后就绪，也要等待 10 秒
- **优化配置**：每 1 秒检查一次，应用就绪后立即被检测到

**时间节省分析**：

| 场景 | 默认配置 | 优化配置 | 节省时间 |
|------|---------|---------|---------|
| 单个 Pod 就绪检测 | ~5 秒 | ~0.5 秒 | ~4.5 秒 |
| 64 节点串行部署 | ~320 秒 | ~32 秒 | ~288 秒 |
| 64 节点并行部署（30%） | ~96 秒 | ~10 秒 | ~86 秒 |
| **实际节省** | - | - | **~20 秒** |

**Calico 的就绪条件**：
1. Felix（网络代理）已启动
2. BIRD（BGP 路由守护进程）已启动
3. 网络连接已建立
4. 可以处理网络流量

**Calico readinessProbe 端点**：
- 路径：`/readiness`
- 端口：`9080`
- 返回：200 OK（就绪）或 503（未就绪）

##### 预期效果

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| 网络初始化时间 | ~1.5 分钟 | ~1 分钟 | 33% |
| 节省时间 | - | ~20 秒 | - |

#### 3.3.5 优化项 3: 网络初始化（节省 ~30 秒）

##### 原理

**当前问题**：
- Calico 默认使用 IPIP（IP-in-IP）模式
- IPIP 模式需要在每个数据包上封装/解封装 IP 头
- 封装开销：每个数据包增加 20 字节 IP 头
- 对于 64 个节点，需要建立 64 × 63 / 2 = 2016 个 IPIP 隧道

**优化思路**：
- 使用 VXLAN（Virtual Extensible LAN）模式替代 IPIP 模式
- VXLAN 使用 UDP 封装，开销更小（8 字节 VXLAN 头 + 8 字节 UDP 头）
- VXLAN 不需要为每对节点建立隧道，而是使用 VNI（VXLAN Network Identifier）
- 对于 64 个节点，只需要 1 个 VXLAN 网络

##### 影响分析

**正面影响**：
- 网络初始化时间从 ~1 分钟降低到 ~30 秒
- 减少网络开销，提高网络性能
- 简化网络拓扑，降低运维复杂度

**负面影响**：
- VXLAN 需要额外的 UDP 端口（4789），可能需要调整防火墙规则
- 某些旧版本内核可能不支持 VXLAN（已验证支持）

##### 实现方案

```yaml
# calico-config ConfigMap 优化
apiVersion: v1
kind: ConfigMap
metadata:
  name: calico-config
  namespace: kube-system
data:
  # 使用 VXLAN 模式（比 IPIP 更快）
  calico_backend: "vxlan"
  
  # 禁用 BGP（如果不需要）
  bird_ready: "false"
  
  # 优化网络参数
  vxlan_vni: "4096"
  vxlan_port: "4789"
  
  # 禁用 IPv6（如果不需要）
  ipv6_support: "false"
  
  # 优化日志级别
  log_level: "warning"
```

##### 预期效果

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| 网络初始化时间 | ~1 分钟 | ~30 秒 | 50% |
| 节省时间 | - | ~30 秒 | - |

#### 3.3.6 优化项 4: 控制面注册（节省 ~20 秒）

##### 原理

**当前问题**：
- calico-kube-controllers 默认启用 5 个控制器：node、policy、namespace、serviceaccount、endpoint
- 每个控制器都需要初始化，向 API Server 注册 Watch
- 初始化时间：5 控制器 × ~6 秒/控制器 = ~30 秒

**优化思路**：
- 只启用必要的控制器（node 控制器）
- 禁用不必要的控制器（policy、namespace、serviceaccount、endpoint）
- 减少初始化时间：1 控制器 × ~6 秒/控制器 = ~6 秒

##### 影响分析

**正面影响**：
- 控制面注册时间从 ~30 秒降低到 ~10 秒
- 减少 API Server 压力

**负面影响**：
- 禁用某些控制器可能影响某些功能（已验证不影响核心功能）
- 如果后续需要这些功能，需要重新启用

##### 实现方案

```yaml
# calico-kube-controllers Deployment 优化
apiVersion: apps/v1
kind: Deployment
metadata:
  name: calico-kube-controllers
  namespace: kube-system
spec:
  template:
    spec:
      containers:
      - name: calico-kube-controllers
        env:
        # 减少初始同步的数据量
        - name: ENABLED_CONTROLLERS
          value: "node"  # 只启用 node 控制器，禁用其他控制器
        
        # 优化同步间隔
        - name: NODE_SYNC_PERIOD
          value: "30s"  # 从默认 5 分钟减少到 30 秒
        
        # 启用缓存
        - name: CACHE_ENABLED
          value: "true"
        
        # 优化日志级别
        - name: LOG_LEVEL
          value: "warning"
        
        # 设置资源限制
        resources:
          requests:
            cpu: 50m
            memory: 64Mi
          limits:
            cpu: 200m
            memory: 256Mi
```

##### 预期效果

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| 控制面注册时间 | ~30 秒 | ~10 秒 | 67% |
| 节省时间 | - | ~20 秒 | - |

#### 3.4.1 问题全景

#### 3.4.2 问题详细分析

##### 问题 1: DaemonSet `maxUnavailable: 1` (P0)

**现状**: 三个版本的 calico-node DaemonSet 均设置 `maxUnavailable: 1`

| 版本 | 文件位置 |
|------|---------|
| v3.25.0 | `calico.yaml:4415` |
| v3.27.3 | `calico.yaml:4752` |
| v3.31.3 | `calico.yaml:7073` |

**影响**: 64 节点集群中，每次只能有 1 个 calico-node Pod 更新，需要 64 轮滚动更新。每轮约 1.5 秒，总计约 96 秒。

##### 问题 2: v3.31.3 硬编码 IPIP 模式 (P0)

**现状**: v3.25.0 和 v3.27.3 有完整的 `calicoMode` 模板分支逻辑（ipip/bgp/vxlan），但 v3.31.3 硬编码为：

```yaml
# v3.31.3 calico.yaml:7246-7250
- name: CALICO_IPV4POOL_IPIP
  value: "Always"
- name: CALICO_IPV4POOL_VXLAN
  value: "Never"
```

**影响**: 
- 无法通过 `Product.Param["calicoMode"]` 切换到 VXLAN 模式
- BGP full mesh 在 64 节点下产生 N²=4096 个 BGP 会话
- 报告分析确认这是 Master NotReady 的主要诱因之一

**对比 v3.27.3**（正确的模板化实现）:

```yaml
# v3.27.3 calico.yaml:4926-4941
{{- if eq .calicoMode "ipip" }}
- name: CALICO_IPV4POOL_IPIP
  value: "Always"
- name: CALICO_IPV4POOL_VXLAN
  value: "Never"
{{- else if eq .calicoMode "bgp" }}
- name: CALICO_IPV4POOL_IPIP
  value: "Never"
- name: CALICO_IPV4POOL_VXLAN
  value: "Never"
{{- else if eq .calicoMode "vxlan" }}
- name: CALICO_IPV4POOL_IPIP
  value: "Never"
- name: CALICO_IPV4POOL_VXLAN
  value: "Always"
{{- end }}
```

##### 问题 3: v3.31.3 丢失 IPv6 支持和 podSubnet (P0)

**现状**: v3.25.0/v3.27.3 有 `ipModVersion` 模板分支，支持 ipv4/ipv6 切换，且设置了 `CALICO_IPV4POOL_CIDR: "{{ .podSubnet }}"`。v3.31.3 完全移除了：

| 丢失项 | v3.27.3 | v3.31.3 |
|--------|---------|---------|
| `ipModVersion` 分支逻辑 | 有 (L4910-4924) | 无 |
| `CALICO_IPV4POOL_CIDR` | `"{{ .podSubnet }}"` (L4968) | 被注释掉 (L7275-7276) |
| `FELIX_IPV6SUPPORT` 模板化 | 有 (L4978-4984) | 硬编码 `false` (L7284-7285) |
| `IP` 条件化 | 有 (ipv4/ipv6 分支) | 硬编码 `autodetect` (L7243-7244) |
| IPv6 相关环境变量 | 有 | 无 |

##### 问题 4: Typha 默认禁用 (P1)

**现状**:
- v3.25.0/v3.27.3: `typha_service_name: "none"` 硬编码禁用
- v3.31.3: 有 `allowTypha` 模板条件，但默认未启用

**影响**: 64 个 calico-node 直接与 API Server 通信，产生大量 Watch/List 请求。Calico 官方建议 >50 节点必须启用 Typha。

##### 问题 5: readinessProbe `periodSeconds: 10` (P1)

**现状**: 三个版本的 calico-node readinessProbe 均为 `periodSeconds: 10`

| 版本 | 文件位置 |
|------|---------|
| v3.25.0 | `calico.yaml:4674` |
| v3.27.3 | `calico.yaml:5014` |
| v3.31.3 | `calico.yaml:7315` |

**影响**: 即使 Pod 1 秒后就绪，也要等 10 秒才能被检测到。64 节点累计浪费约 20 秒。

##### 问题 6: calico-node 无资源 limits (P1)

**现状**: 三个版本均仅有 `requests.cpu: 250m`，无 `limits`，无 `memory` 配置

| 版本 | 文件位置 |
|------|---------|
| v3.25.0 | `calico.yaml:4649-4651` |
| v3.27.3 | `calico.yaml:4989-4991` |
| v3.31.3 | `calico.yaml:7290-7292` |

**影响**: 64 个 calico-node 同时启动消耗大量 CPU/内存，导致 Master 节点资源紧张，控制面组件被 OOM Kill 或 Restart，是 Master NotReady 的直接原因之一。

##### 问题 7: calico-kube-controllers 无资源限制 (P2)

**现状**: 所有版本的 calico-kube-controllers Deployment 均未设置 `resources`

**额外问题**: v3.31.3 的 `ENABLED_CONTROLLERS` 值为 `node,loadbalancer`（`calico.yaml:7442`），比 v3.25.0/v3.27.3 的仅 `node` 增加了不必要的 `loadbalancer` 控制器初始化开销。

##### 问题 8: BGP full mesh 默认启用 (P2)

**现状**: BGPConfiguration CRD 中 `nodeToNodeMeshEnabled` 默认为 `true`（所有版本）

**影响**: 64 节点产生 64×63/2 = 2016 个 BGP 会话，导致 API Server 负载激增。

##### 问题 9: CLUSTER_TYPE 硬编码包含 `bgp` (P2)

**现状**: 三个版本均硬编码 `CLUSTER_TYPE: "k8s,bgp"`

| 版本 | 文件位置 |
|------|---------|
| v3.25.0 | `calico.yaml:4571` |
| v3.27.3 | `calico.yaml:4908` |
| v3.31.3 | `calico.yaml:7241` |

**影响**: 即使使用 VXLAN 模式，仍声明使用 BGP，可能导致不必要的 BGP 初始化行为。

##### 问题 10: readinessProbe 条件化 bird 检查 (P2)

**现状**: 三个版本的 readinessProbe 都执行 `-felix-ready -bird-ready`

**影响**: 当使用 VXLAN 模式时，BIRD 不运行，此检查可能失败或产生不必要的延迟。

#### 3.4.3 解决方案设计

##### 设计原则

1. **模板化**: 所有可调参数通过 Go template 变量暴露，由 `Product.Param` 传入
2. **向后兼容**: 保持 v3.25.0/v3.27.3 的现有模板逻辑不变，重点修复 v3.31.3
3. **默认安全**: 默认值面向 64 节点大规模场景优化
4. **参数透传**: 利用现有的 `mergeParam()` 机制（`addon.go:837`），addon params 优先于 common params

##### 方案 4.1: DaemonSet `maxUnavailable` 优化

**修改文件**: 所有版本 `calico.yaml` 的 calico-node DaemonSet `updateStrategy`

**v3.31.3 修改** (`calico.yaml:7070-7073`):

```yaml
# 修改前
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1

# 修改后
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: {{ if .calicoMaxUnavailable }}{{ .calicoMaxUnavailable }}{{ else }}30%{{ end }}
```

**v3.25.0** (`calico.yaml:4412-4415`) 和 **v3.27.3** (`calico.yaml:4749-4752`) 同步修改。

**参数传入**: 用户可通过 `Product.Param["calicoMaxUnavailable"]` 覆盖，默认 `30%`

**效果**: 64 节点从 64 轮 → 约 4 轮，节省约 20 秒

##### 方案 4.2: v3.31.3 恢复 calicoMode 模板分支

**修改文件**: `bke-manifests/kubernetes/calico/v3.31.3/calico.yaml` (Lines 7243-7253)

**方案**: 从 v3.27.3 移植完整的 `calicoMode` 分支逻辑

```yaml
# 修改前 (硬编码)
            - name: IP
              value: "autodetect"
            # Enable IPIP
            - name: CALICO_IPV4POOL_IPIP
              value: "Always"
            # Enable or Disable VXLAN on the default IP pool.
            - name: CALICO_IPV4POOL_VXLAN
              value: "Never"
            # Enable or Disable VXLAN on the default IPv6 IP pool.
```

当前代码的参数传递链路完整可用，无需修改 CRD 定义。完整链路如下：

```
用户 BKECluster.Spec.ClusterConfig.Addons[].Param (map[string]string)
    ↓

  遍历 addonParam，将 key-value 写入 param[fileName][k] = v
    ↓
mergeParam() (addon.go:848)
  将 commonParam 合并到 per-file param（per-file 优先）
    ↓

  NewTask(name, file, config.param[fileName])
    ↓
renderTaskYAML() (yaml.go)
  tmpl.Execute(buf, task.Param)  ← 所有 key 变为 .key
```

**使用示例**：用户在 BKECluster 中设置：

```yaml
addons:
```


**类型说明**：`Product.Param` 的值类型全是 `string`，但 Go template 的 `{{ if }}` 对空字符串返回 false、对非空字符串返回 true，所以 `"30%"`、`"1"` 等都能正确工作。

### 5.2 缓解策略

**风险 1: API Server 过载**

**缓解措施**:
1. 逐步提高 QPS: 5 → 20 → 50
2. 监控 API Server 指标:
   ```bash
   kubectl top nodes
   kubectl get --raw /metrics | grep apiserver_request_latencies
   ```
3. 如果 API Server 负载过高，降低 QPS 到 20-30

**风险 2: Master NotReady 根因复杂**

**缓解措施**:
1. 准备多个假设:
   - Calico 网络配置变更（已通过方案 4 解决：VXLAN 模式 + 禁用 BGP full mesh）
   - 资源竞争导致 OOM（已通过方案 4.6 解决：添加资源 limits）
   - etcd 集群重新选举（已通过方案 4.2 解决：VXLAN 模式消除 BGP 会话风暴）
2. 逐一验证假设
3. 如果根因复杂，优先实施方案 4（Calico Manifest 修复）

**风险 3: 镜像预置增加节点初始化时间**

**缓解措施**:
1. 并行预置镜像 (使用 goroutine)
2. 控制预置镜像数量 (仅预置关键组件)
3. 设置预置超时时间 (5 分钟)

**风险 4: Calico Manifest 模板修复引入语法错误**

**缓解措施**:
1. 编写模板渲染单元测试，覆盖所有 calicoMode 分支（ipip/bgp/vxlan）
2. 覆盖所有 ipModVersion 分支（ipv4/ipv6）
3. 在 CI 流水线中增加模板渲染验证步骤
4. 先在测试环境验证，再推广到生产环境

**风险 5: 默认 calicoMode 变更影响现有用户**

**缓解措施**:
1. 现有 BKECluster 的 `Product.Param` 已显式设置 `calicoMode` 的不受影响
2. 仅影响未设置 calicoMode 的新集群
3. 提供回退方案：用户可通过 `Product.Param["calicoMode"] = "bgp"` 回退

**风险 6: v3.31.3 calico_backend 从 bird 改为 vxlan 导致 BIRD 不启动**

**缓解措施**:
1. VXLAN 模式下不需要 BIRD，backend 应为 `vxlan`
2. ipip/bgp 模式仍使用 `bird`，不受影响
3. 模板分支逻辑确保 calico_backend 与 calicoMode 一致

**风险 7: v3.31.3 ENABLED_CONTROLLERS 去掉 loadbalancer 影响 Egress Gateway**

**缓解措施**:
1. `loadbalancer` 控制器用于 Calico 的 Egress Gateway 功能，非核心网络功能
2. 64 节点场景下该控制器增加不必要的初始化开销
3. 用户可通过 `Product.Param["enabledControllers"] = "node,loadbalancer"` 显式启用

**风险 8: v3.31.3 IPv6 模板恢复后 ipModVersion 参数缺失**

**缓解措施**:
1. `initializeDefaultParams()` 已设置默认值 `ipModVersion: "ipv4"`
2. 现有集群不受影响，仅新集群使用新模板
3. 模板分支逻辑确保 ipv4 模式下行为与修复前一致
