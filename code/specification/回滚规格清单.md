# 回滚规格清单
          
下面是对代码中实际实现的回滚规格的梳理。

## 代码中已实现的回滚规格梳理

工程中的回滚机制分散在多个层次，整体遵循"分级回滚"思路，但实现程度不一。下面按层次梳理。

### 一、声明式 DAG 层的失败策略（已定义，部分实现）

**位置**：[pkg/topology/component.go:22-27](file:///cluster-api-provider-bke/pkg/topology/component.go#L22-L27)
```go
const (
    FailurePolicyFailFast FailurePolicy = "FailFast"
    FailurePolicyContinue FailurePolicy = "Continue"
    FailurePolicyRollback FailurePolicy = "Rollback"
)
```
**实际实现状态**（[pkg/dagexec/scheduler.go:189-243](file:///cluster-api-provider-bke/pkg/dagexec/scheduler.go#L189-L243)）：

| 策略 | 设计意图 | 代码实现 |
|------|----------|----------|
| `FailFast` | 失败即停止整个 DAG | ✅ 已实现：`failFastStop = true`，中断后续批次 |
| `Continue` | 失败继续下一批次 | ✅ 已实现：错误收集但不停止 |
| `Rollback` | 失败后回滚到旧版本 | ❌ **仅定义常量，scheduler 中无任何分支处理** |

**关键缺口**：`FailurePolicyRollback` 在 `executeBatchParallel` 和 `persistBatchResults` 中都没有对应的处理分支，仅与 `FailFast` 做等价处理。真正的回滚动作（执行 uninstallScript / helm rollback）未实现。

### 二、副本数回滚（扩缩容场景，已实现）

这是工程中**唯一完整实现**的回滚类型，通过 `defer` + 副本数恢复实现。

#### 2.1 Worker 加入回滚

**位置**：[pkg/phaseframe/phases/ensure_worker_join.go:205-213](file:////cluster-api-provider-bke/pkg/phaseframe/phases/ensure_worker_join.go#L205-L213)
```go
defer func() {
    if scaleErr != nil {
        log.Info(constant.WorkerJoinFailedReason, "Scale down MachineDeployment replicas to %d.", *currentReplicas)
        params.Scope.MachineDeployment.Spec.Replicas = currentReplicas
        if err := phaseutil.ResumeClusterAPIObj(...); err != nil {
            log.Error(constant.WorkerJoinFailedReason, "Rollback MachineDeployment replicas failed. err: %v", err)
        }
    }
}()
```
**规格**：
- 触发条件：`scaleErr != nil`
- 回滚动作：将 `MachineDeployment.Spec.Replicas` 恢复到扩容前的值
- 失败处理：仅日志记录，不重试

#### 2.2 Master 加入回滚

**规格**：同上，针对 `KubeadmControlPlane.Spec.Replicas`。

#### 2.3 Worker 删除回滚

#### 2.4 Master 删除回滚

**位置**：[pkg/phaseframe/phases/ensure_master_delete.go:187-195](file:///cluster-api-provider-bke/pkg/phaseframe/phases/ensure_master_delete.go#L187-L195)

**副本数回滚的共同规格**：

| 维度 | 规格 |
|------|------|
| 触发条件 | 函数返回 error（通过 `defer` + 闭包变量捕获） |
| 回滚动作 | 恢复 `Spec.Replicas` 到操作前值 + `ResumeClusterAPIObj` |
| 失败处理 | 仅 `log.Error`，不向上抛出 |
| 范围 | 单次 reconcile 内有效，不跨周期 |

### 四、Containerd 升级的"重置+重部署"模式（非真正回滚）

**位置**：[pkg/phaseframe/phases/ensure_containerd_upgrade.go:100-160](file:///cluster-api-provider-bke/pkg/phaseframe/phases/ensure_containerd_upgrade.go#L100-L160)
```go
func (e *EnsureContainerdUpgrade) rolloutContainerd() (ctrl.Result, error) {
    if err := e.resetContainerd(); err != nil { return ... }
    if err := e.redeployContainerd(); err != nil { return ... }
    bkeCluster.Status.ContainerdVersion = targetVersion
    return ...
}
```
**规格**：
- 模式：`reset` → `redeploy`（两步原子替换，非回滚）
- 失败行为：reset 失败或 redeploy 失败均直接返回 error，**不会恢复到旧版本**
- 中间状态：reset 成功但 redeploy 失败时，节点处于无 containerd 状态，需人工介入

**这是工程中的设计盲点**：containerd 升级失败后没有回滚到旧版本 containerd 的能力。

### 十、关键结论

1. **副本数回滚是唯一完整自动化的回滚**：覆盖 Master/Worker 的 Join/Delete，但仅限 `Spec.Replicas` 字段。
2. **备份机制完整但恢复路径缺失**：etcd snapshot、/etc/kubernetes、配置文件、bkeagent 二进制均有备份，但全部依赖人工救援，代码中无 `Restore` 调用。
3. **DAG 层 Rollback 策略是空壳**：`FailurePolicyRollback` 常量已定义，但 scheduler 中无任何分支处理，与 `FailFast` 行为一致。
4. **组件升级失败无版本回滚**：containerd/master/worker 升级失败后仅标记节点状态，不回滚到旧版本，节点停留在中间状态。
5. **KEP-6 设计了完整回滚规格但未落地**：`RollbackSpec`、`UninstallSpec`、`HelmActionRollback`、`--atomic` 均停留在设计文档。
