## 二、Phase 状态机：6 种状态 × 26 个 Phase × 10+ 种集群场景

### 2.1 单个 Phase 状态机

```
                    ┌─────────────┐
                    │   Waiting   │ ← 初始状态，等待执行
                    └──────┬──────┘
                           │ NeedExecute() = true
                           ▼
                    ┌─────────────┐
                    │   Running   │ ← 正在执行
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
       ┌──────────┐ ┌──────────┐ ┌──────────┐
       │Succeeded │ │  Failed  │ │ Skipped  │
       │   ✅     │ │   ❌     │ │   ⏭️     │
       └──────────┘ └──────────┘ └──────────┘
                            │
                            ▼  (Failed 终止整个流水线)
                     本次调谐结束，等待下次重试
```

### 2.2 复杂度矩阵：状态 × 场景 × Phase 分组

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  集群场景 (10+)           │ 涉及的 Phase 分组      │ 状态映射逻辑            │
├───────────────────────────┼────────────────────────┼─────────────────────────┤
│  初始化 (Initializing)    │ ClusterInitPhaseNames  │ Running→Initializing    │
│                           │ (8个Phase)             │ Succeeded→继续           │
│                           │                        │ Failed→InitFailed       │
├───────────────────────────┼────────────────────────┼─────────────────────────┤
│  升级 (Upgrading)         │ ClusterUpgradePhase    │ Running→Upgrading       │
│                           │ (5个Phase)             │ Succeeded→继续           │
│                           │                        │ Failed→UpgradeFailed    │
├───────────────────────────┼────────────────────────┼─────────────────────────┤
│  Master 扩容             │ ClusterScaleMasterUp   │ Running→ScalingMasterUp │
│  Master 缩容             │ ClusterScaleMasterDown │ ...                     │
│  Worker 扩容             │ ClusterScaleWorkerUp   │                         │
│  Worker 缩容             │ ClusterScaleWorkerDown │                         │
├───────────────────────────┼────────────────────────┼─────────────────────────┤
│  组件部署 (DeployingAddon)│ ClusterAddonsPhase     │ Running→DeployingAddon  │

│  暂停 (Paused)            │ ClusterPausedPhase     │ Running→Paused          │

├───────────────────────────┼────────────────────────┼─────────────────────────┤

│  (EnsureCluster)          │ (1个Phase)             │ 内部判断 Ready/Unhealthy│
└───────────────────────────┴────────────────────────┴─────────────────────────┘
```

## 三、当前架构的 7 大复杂度痛点

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  痛点 1: 依赖关系隐式化 — 靠代码顺序，无法并行                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  当前:  ①→②→③→④→⑤→⑥→⑦→⑧→⑨→⑩→⑪→⑫→⑬→⑭→⑮→⑯→⑰→⑱→⑲→⑳→㉑→㉒→㉓→㉔→㉕           │
│         (全部串行，即使 ⑧和⑨ 无依赖也必须等)                                  │
│                                                                              │
│  实际依赖关系 (部分):                                                          │
│                                                                              │
│         EnsureMasterInit ──▶ EnsureMasterJoin ──▶ EnsureWorkerJoin          │
│                  ▲                                      │                    │
│                  │                                      ▼                    │
│         EnsureCerts ──▶ EnsureLoadBalance         EnsureAddonDeploy          │
│                                                                              │
│  问题: ⑧Certs 和 ⑨LoadBalance 可以并行，但当前只能串行                         │
│        升级场景下 5 个升级 Phase 中 3 个可并行，被强制串行                     │
│                                                                              │
├─────────────────────────────────────────────────────────────────────────────┤
│  痛点 2: 新增 Phase 需修改 3 处以上代码                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  新增一个 Phase "EnsureXXX" 需要:                                             │
│  ┌─────────────────────────────────────────────────────────────────┐        │
│  │  ① list.go: 添加到 CommonPhases/DeployPhases/PostDeployPhases   │        │
│  │  ② list.go: 添加到对应的 ClusterXXXPhaseNames 分组              │        │
│  │  ③ phase_flow.go: calculateClusterStatusByPhase() 加 case 分支  │        │
│  │  ④ 实现 NeedExecute() 中的复杂条件判断                           │        │
│  │  ⑤ PhaseNameCNMap 加中文映射                                    │        │
│  └─────────────────────────────────────────────────────────────────┘        │
│                                                                              │
│  影响: 新人不敢改，老人容易漏改，测试覆盖难                                   │
│                                                                              │
├─────────────────────────────────────────────────────────────────────────────┤
│  痛点 3: 安装/升级/删除 代码混用 — 26 个 Phase 一套流水线                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  当前: 每次调谐遍历全部 26 个 Phase，每个 Phase 内部判断场景                   │
│                                                                              │
│  EnsureWorkerUpgrade.NeedExecute():                                           │
│  ┌─────────────────────────────────────────────────────────────────┐        │
│  │  if 集群在删除 → return false                                    │        │
│  │  if 集群在初始化 → return false                                  │        │
│  │  if Worker 版本 == 目标版本 → return false (Skipped)             │        │
│  │  if 控制面未就绪 → return false                                  │        │
│  │  if 正在升级 → return false                                      │        │
│  │  ... 8+ 个条件判断                                               │        │
│  └─────────────────────────────────────────────────────────────────┘        │
│                                                                              │
│  问题: 安装场景下 20 个 Phase 被 Skipped，升级场景下 15 个被 Skipped           │
│        大量无效遍历，逻辑分散在各 Phase 内部                                  │
│                                                                              │
├─────────────────────────────────────────────────────────────────────────────┤
│  痛点 4: 状态映射逻辑硬编码 — 10+ 个 case 分支                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  calculateClusterStatusByPhase():                                             │
│  ┌─────────────────────────────────────────────────────────────────┐        │
│  │  switch {                                                       │        │
│  │  case phaseName.In(CustomSetStatusPhaseNames): return nil       │        │
│  │  case phaseName.In(ClusterInitPhaseNames):                      │        │
│  │      if err != nil { status = InitializationFailed }            │        │
│  │      else { status = Initializing }                             │        │
│  │  case phaseName.In(ClusterUpgradePhaseNames): ...               │        │
│  │  case phaseName.In(ClusterScaleMasterUpPhaseNames): ...         │        │
│  │  case phaseName.In(ClusterScaleMasterDownPhaseNames): ...       │        │
│  │  case phaseName.In(ClusterScaleWorkerUpPhaseNames): ...         │        │
│  │  case phaseName.In(ClusterScaleWorkerDownPhaseNames): ...       │        │
│  │  case phaseName.In(ClusterAddonsPhaseNames): ...                │        │
│  │  case phaseName.In(ClusterDeletePhaseNames): ...                │        │
│  │  case phaseName.In(ClusterDryRunPhaseNames): ...                │        │
│  │  case phaseName.In(ClusterPausedPhaseNames): ...                │        │
│  │  case phaseName.In(ClusterManagePhaseNames): ...                │        │
│  │  default: status = Unknown                                      │        │
│  │  }                                                              │        │
│  └─────────────────────────────────────────────────────────────────┘        │
│                                                                              │
│  问题: 每新增一个 Phase 分组，就要加一个 case 分支                            │
│                                                                              │
├─────────────────────────────────────────────────────────────────────────────┤
│  痛点 5: 失败策略单一 — 失败即终止，无回滚/跳过/继续选项                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  当前: 任何 Phase 失败 → return (终止整个流水线) → 等待下次调谐重试           │
│                                                                              │
│  缺失:                                                                        │
│  • FailFast: 核心组件失败立即终止 (当前默认行为)                              │
│  • Continue: 非核心组件失败跳过继续 (如监控组件)                              │
│  • Rollback: 核心组件失败触发回滚 (如 etcd 升级失败)                          │
│                                                                              │
├─────────────────────────────────────────────────────────────────────────────┤
│  痛点 6: 状态追踪不透明 — 只有成功/失败，无进度预估                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  PhaseStatus 只记录: Name, StartTime, EndTime, Status, Message               │
│                                                                              │
│  缺失:                                                                        │
│  • 当前执行到第几步 / 总共几步                                                │
│  • 预计剩余时间                                                               │
│  • 哪些 Phase 被 Skipped 及原因                                               │
│  • 升级路径可视化 (多跳升级时)                                                │
│                                                                              │
├─────────────────────────────────────────────────────────────────────────────┤
│  痛点 7: NeedExecute 逻辑重复 — 每个 Phase 都写一遍                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  26 个 Phase 中，每个都实现了 NeedExecute():                                  │
│  • 8 个公共检查 (删除中? 暂停? DryRun? 健康? ...) — 代码重复 26 次            │
│  • 各 Phase 特有检查 — 散落在 26 个文件中                                     │
│  • 修改公共检查逻辑 → 需要改 26 个文件                                        │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 六、重构后 Phase 决策机制

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    重构后: Phase 升级决策 (复用 NeedExecute)                  │
│                                                                              │
│  旧方案: 每个 Phase 内部写 8+ 个条件判断                                     │
│  新方案: 通过版本上下文自动判断                                                │
│                                                                              │
│  ┌─────────────────────────────────────────────────────────────────┐        │
│  │                                                                 │        │
│  │  1. 从 ClusterVersion 找到目标 ReleaseImage                     │        │
│  │     → 目标组件列表: [(k8s, v1.29.0), (etcd, v3.5.12), ...]     │        │
│  │                                                                 │        │
│  │  2. 从 ClusterVersion 找到当前 ReleaseImage                     │        │
│  │     → 当前组件列表: [(k8s, v1.28.0), (etcd, v3.5.10), ...]     │        │
│  │                                                                 │        │
│  │  3. 对比版本:                                                    │        │
│  │     NeedExecute() = 目标版本 != 当前版本                         │        │
│  │                                                                 │        │
│  │     k8s:  v1.29.0 != v1.28.0 → 需要升级 ✅                      │        │
│  │     etcd: v3.5.12 != v3.5.10 → 需要升级 ✅                      │        │
│  │     agent: v2.6.0 == v2.6.0 → 跳过 ⏭️                          │        │
│  │                                                                 │        │
│  │  4. 执行升级:                                                    │        │
│  │     调用 Phase.Execute() (核心业务逻辑不变)                      │        │
│  │                                                                 │        │
│  └─────────────────────────────────────────────────────────────────┘        │
│                                                                              │
│  收益:                                                                        │
│  • 代码分支减少 70%                                                           │
│  • 历史 Bug 重复率降低 80%                                                    │
│  • 测试工作量减半                                                             │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```
